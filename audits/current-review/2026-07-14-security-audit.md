# Полный аудит безопасности We Designerz / VibeClub

**Дата проверки:** 14 июля 2026  
**Статус:** запуск на широкую аудиторию не рекомендован до закрытия P0 и P1  
**Тип проверки:** защитный аудит собственного проекта, без разрушающих действий

## 1. Итог для владельца

Платформа пока не готова к резкому притоку пользователей после публичного видео.
Базовая защита уже есть: RLS включён, `service_role` не публикуется, внутренние
порты VPS закрыты, TLS настроен, бэкапы выполняются, а пользовательский текст в
основном выводится безопасно. Но аудит подтвердил две критические проблемы и
несколько серьёзных путей для злоупотреблений.

### Критические проблемы

1. **Активный GitHub Personal Access Token находится в URL локального `origin`.**
   Токен не закоммичен в историю, но он действующий и уже был показан инструменту
   аудита. Его нужно немедленно отозвать и выпустить новый с минимальными правами.
2. **Владелец проекта может напрямую установить себе любое число голосов.**
   Это подтверждено безопасным транзакционным тестом на self-hosted БД: значение
   `projects.upvotes` удалось изменить от имени обычного пользователя. После теста
   транзакция была отменена, тестовая строка в базе не осталась.

### Решение о запуске

- **P0:** закрыть до любых анонсов и привлечения трафика.
- **P1:** закрыть до публикации видео или включить временные ограничения,
  перечисленные в разделе 6.
- **P2:** допустимо исправлять после ограниченного закрытого запуска, но с
  назначенными сроками.

Аудит не может обещать «абсолютную невзламываемость». После выполнения плана нужен
повторный прогон и независимый ручной pentest перед действительно крупным запуском.

## 2. Что проверено

- публичный сайт `wedesignerz.com` и его HTTP-заголовки;
- публичные API cloud Supabase и self-hosted Supabase;
- Supabase Auth, REST/RPC, Storage, схемы, RLS-политики, функции и права;
- клиентский HTML/CSS/JS: XSS, внешние URL, загрузка файлов, auth-потоки;
- GitHub Pages workflow и состав публикуемого артефакта;
- локальный Git-репозиторий и история на типовые секреты;
- VPS: SSH, firewall, открытые порты, Docker Compose, Caddy и контейнеры;
- бэкапы БД, Storage и Umami;
- npm-зависимости, линтеры и существующий security-check;
- образы всех запущенных контейнеров сканером Trivy.

### Что намеренно не делалось

- DDoS, нагрузочные атаки, подбор паролей и социальная инженерия;
- удаление или изменение рабочих данных;
- эксплуатация уязвимостей контейнеров;
- проверка чужих аккаунтов;
- отправка вредоносных файлов реальным пользователям.

Единственный тест записи выполнен внутри транзакции PostgreSQL с последующим
`ROLLBACK`; отсутствие тестовых данных проверено отдельно.

## 3. Карта поверхности атаки

Сейчас доступны сразу два Supabase-бэкенда:

1. **Cloud Supabase** — именно его использует production-фронтенд из `js/config.js`.
2. **Self-hosted Supabase** — доступен через `api.wedesignerz.com`, но фронтенд на
   него ещё не переключён.

Это удваивает поверхность атаки и создаёт расхождение схем, миграций, настроек Auth
и политик. Нужно выбрать один production-бэкенд, перенести данные, проверить его и
закрыть или жёстко ограничить второй.

## 5. Подробности и исправления

### SEC-01 — действующий GitHub-токен в remote URL

**Доказательство.** В `.git/config` адрес `origin` содержит classic PAT. Проверка
доступа к репозиторию с этим remote прошла успешно. В отслеживаемых файлах и истории
Git токен не найден, поэтому проблема локальная, но токен всё равно считается
скомпрометированным.

**Что может случиться.** В зависимости от выданных scopes злоумышленник сможет
читать/изменять репозиторий, workflows, Pages или другие ресурсы аккаунта.

**Исправление владельцем:**

1. На GitHub открыть **Settings → Developer settings → Personal access tokens**.
2. Немедленно отозвать найденный classic token.
3. Заменить remote на адрес без секрета или SSH URL.
4. Если токен ещё нужен — создать fine-grained token только для этого репозитория,
   с минимальным сроком и минимальными правами.
5. Проверить GitHub audit/security log на неизвестные операции.

**Критерий закрытия:** `git remote -v` не содержит секретов; старый токен отозван;
push работает через SSH или системный credential manager.

### SEC-02 — подтверждённая накрутка `upvotes`

**Причина.** Политика `projects_update_own` разрешает владельцу обновлять всю строку
проекта (`supabase/schema.sql:356-362`). Триггер защищает только `status` и `is_core`
(`supabase/schema.sql:232-243`), но не серверный счётчик `upvotes`.

**Доказательство.** В транзакции создан тестовый проект, затем под JWT обычного
участника выполнено изменение `upvotes` на произвольное значение. UPDATE прошёл.
После этого выполнен `ROLLBACK`, число оставшихся тестовых строк — ноль.

**Что исправить:**

- запретить клиентским ролям UPDATE служебных колонок;
- разрешать автору менять только явный набор полей: название, описание, URL,
  обложку, теги, инструменты и навыки;
- `upvotes` изменять исключительно триггером из `project_upvotes`;
- дополнительно защищать `created_at`, `author_id`, `status`, `is_core`;
- добавить regression-тест: попытка владельца изменить каждое служебное поле
  должна завершаться отказом.

Предпочтительный вариант — отозвать общий UPDATE у `authenticated` и выдать права
на конкретные колонки, плюс оставить защитный trigger как второй слой.

**Критерий закрытия:** прямой REST/SQL UPDATE `upvotes` обычным пользователем получает
403/permission error; обычное редактирование текста проекта продолжает работать;
голос меняется только после INSERT/DELETE в `project_upvotes`.

### SEC-03 — функции, `PUBLIC` и default privileges

В обеих БД API-роли получают широкие привилегии по умолчанию. Несколько
`SECURITY DEFINER`-функций имеют EXECUTE для `PUBLIC`, `anon` и `authenticated`.
Миграции отзывают право только у двух ролей
(`supabase/schema.sql:301-310` и миграция `2026-07-11-antispam-limits.sql:124-150`),
но право через `PUBLIC` остаётся. Поэтому комментарий «перестанут торчать в RPC»
не соответствует реальному состоянию.

Прямой HTTP-вызов большинства trigger-функций сейчас не прошёл из-за их сигнатуры,
то есть это не подтверждённое удалённое выполнение кода. Однако это опасная модель
прав: любое изменение сигнатуры или новая функция может автоматически открыть
привилегированную операцию через PostgREST.

**Что исправить:**

- `REVOKE EXECUTE ... FROM PUBLIC, anon, authenticated` для внутренних функций;
- перенести trigger/helper-функции в закрытую схему `private`;
- ставить `search_path = ''` и квалифицировать все таблицы схемой;
- оставить публичным только строго необходимый RPC `is_admin`, либо заменить его
  приватной проверкой в политиках;
- отозвать default privileges для таблиц, sequences и functions, затем выдавать
  доступ явно для каждого объекта;
- исправить mutable `search_path` у `array_elems_fit`.

Руководство: [Supabase — Database Functions](https://supabase.com/docs/guides/database/functions),
[Supabase — Securing your API](https://supabase.com/docs/guides/api/securing-your-api).

### SEC-04 и SEC-18 — Storage принимает произвольный контент и позволяет листинг

Bucket `covers` публичный, а в БД у него отсутствуют `file_size_limit` и
`allowed_mime_types`. Анонимный запрос списка объектов успешно вернул содержимое
bucket как в cloud, так и в self-hosted Supabase.

Проверка JPEG/PNG/WebP и лимита 10 МБ есть только в браузере. Прямой API-запрос её
обходит. Зарегистрированный пользователь может загружать файлы в свою папку, но
без серверного MIME-лимита, пользовательской квоты и нормального rate limit.

**Что исправить:**

- задать bucket-level `file_size_limit` не более 10 МБ;
- разрешить только `image/jpeg`, `image/png`, `image/webp`;
- после загрузки декодировать изображение сервером, удалять метаданные и сохранять
  безопасную производную версию;
- ограничить число файлов и общий объём на пользователя;
- удалить broad SELECT policy для листинга, если публичные ссылки уже работают без
  неё; проверить чтение конкретного объекта после изменения;
- добавить удаление сиротских файлов и антиабуз-мониторинг.

Руководство: [Supabase Storage access control](https://supabase.com/docs/guides/storage/security/access-control).

### SEC-05 — анонимный feedback можно спамить напрямую

Политика разрешает INSERT для `anon` (`supabase/schema.sql:416-424`). Honeypot и
60-секундный cooldown находятся в JS (`js/ui/feedbackModal.js:137-169`) и не
защищают REST endpoint.

**Что исправить:**

- убрать прямой анонимный INSERT в таблицу;
- принимать форму через Edge Function или Cloudflare Worker;
- проверять Turnstile/CAPTCHA на сервере;
- добавить лимиты по IP, fingerprint/session и дневной общий предел;
- ограничить максимальный размер тела до передачи в БД;
- логировать всплески и предусмотреть аварийное отключение формы.

**Временная безопасная мера:** до появления server-side gateway разрешить feedback
только авторизованным пользователям.

### SEC-06 и SEC-07 — заголовки браузера и CDN-код

Основной сайт на GitHub Pages не отдаёт CSP, HSTS, `X-Content-Type-Options`,
`Referrer-Policy`, `Permissions-Policy` и защиту от встраивания во frame. При этом
Supabase-клиент импортируется с `https://esm.sh/@supabase/supabase-js@2`
(`js/supabase.js:5`) — это плавающая major-версия удалённого исполняемого кода без
локального хеша. Компрометация CDN или неожиданное обновление получает доступ к
сессии пользователя на origin сайта.

**Что исправить:**

- скачать и хранить точную проверенную версию Supabase JS внутри репозитория;
- обновлять её контролируемо через PR, lockfile и security scan;
- перенести основной сайт за CDN/proxy, позволяющий задавать response headers;
- внедрить строгий CSP без `unsafe-inline` и `unsafe-eval`;
- вынести оставшиеся inline scripts в `js/`;
- добавить HSTS на apex-домен, `nosniff`, `frame-ancestors 'none'`, строгий
  Referrer-Policy и минимальный Permissions-Policy;
- ограничить `connect-src`, `img-src` и `script-src` реальными доменами.

Meta-CSP допустим только как временная мера: он не заменяет все HTTP-заголовки и
не поддерживает полноценный `frame-ancestors`.

### SEC-08 — известные уязвимости в Docker images

Trivy 0.70.0 просканировал все запущенные образы с режимом
`HIGH,CRITICAL --ignore-unfixed`. Найдено **14 Critical и 277 High package findings**.
Это количество совпадений пакетов по образам, а не число уникальных эксплуатируемых
уязвимостей. Сканер сообщил, что доступна версия 0.72.0, поэтому после обновления
нужен повторный прогон.

| Образ | Critical | High |
|---|---:|---:|
| Caddy | 6 | 60 |
| imgproxy | 2 | 35 |
| Umami 2.20.2 | 3 | 76 |
| Kong | 0 | 3 |
| PostgREST | 0 | 0 |
| GoTrue | 1 | 29 |
| postgres-meta | 2 | 23 |
| Supabase Postgres | 0 | 15 |
| Storage API | 0 | 36 |

Основные повторяющиеся группы затрагивают OpenSSL/libcrypto/libssl, GnuTLS,
Go crypto/telemetry, Node/gRPC и системные пакеты. Наличие CVE не доказывает, что
она достижима через текущую конфигурацию, но исправляемые Critical findings в
публичных образах нельзя оставлять перед запуском.

**Что исправить:**

1. Обновить self-hosted Supabase по официальному Docker guide и changelog.
2. Обновить Caddy, imgproxy, Umami и остальные образы до исправленных релизов.
3. Закрепить проверенные образы по digest, а не только по tag.
4. Сначала проверить обновление на staging и протестировать Auth, Storage, REST,
   RLS, backup/restore и rollback.
5. Повторить актуальным Trivy; launch gate — 0 исправляемых Critical. Для High
   требуется либо обновление, либо письменное обоснование недостижимости.

Источники: [официальный Docker guide Supabase](https://github.com/supabase/supabase/blob/master/docker/README.md),
[релизы Supabase](https://github.com/supabase/supabase/releases),
[релизы Umami](https://github.com/umami-software/umami/releases),
[релизы Trivy](https://github.com/aquasecurity/trivy/releases).

### SEC-09 — SSH hardening фактически не применился полностью

Документ `docs/15-security-hardening.md:13-18` утверждает, что password auth
отключён. Но итоговая конфигурация `sshd -T` возвращает
`PasswordAuthentication yes`: `50-cloud-init.conf` успевает задать значение раньше
`99-hardening.conf`. OpenSSH использует первое применимое значение.

Сейчас `PermitRootLogin prohibit-password`, а root — единственный shell-account,
поэтому парольный вход root отбивается. Это уменьшает немедленный риск, но защита
хрупкая: любой добавленный пользователь с паролем окажется доступен для перебора.
За 24 часа в журнале было 188 неудачных паролей и 114 попыток для несуществующих
пользователей. fail2ban отсутствует.

**Что исправить:**

- перенести hardening в файл, который читается раньше cloud-init, например
  `00-hardening.conf`, либо изменить cloud-init-конфигурацию;
- оставить `PasswordAuthentication no`, `KbdInteractiveAuthentication no`,
  `PermitRootLogin prohibit-password`;
- уменьшить `MaxAuthTries`, `LoginGraceTime`, при возможности задать `AllowUsers`;
- установить fail2ban/CrowdSec для SSH и публичных HTTP endpoint;
- проверить синтаксис `sshd -t`, не закрывая текущую сессию;
- открыть вторую SSH-сессию по ключу и только затем перезагрузить sshd;
- подтвердить результат именно через `sshd -T`.

### SEC-10 — защита Auth и действий от ферм аккаунтов

Регистрация открыта, email-подтверждение включено, а anonymous sign-in выключен —
это хорошо. Но leaked-password protection отключена, CAPTCHA/Turnstile не
подтверждена, а серверного лимита числа проектов на аккаунт нет. Один голос на
аккаунт не мешает ферме создавать множество подтверждённых аккаунтов.

**Что исправить:**

- включить leaked-password protection;
- минимальная длина пароля — 12 символов;
- включить CAPTCHA/Turnstile для signup, password reset и magic-link;
- проверить и ужесточить Auth rate limits;
- ввести лимит pending-проектов и суточных публикаций на аккаунт;
- для голосования учитывать возраст/подтверждение аккаунта и детектировать всплески;
- добавить ручной kill switch для регистрации, feedback и submission.

Руководство: [Supabase product security](https://supabase.com/docs/guides/security/product-security).

### SEC-11 — другие служебные поля доступны через широкий UPDATE

Проблема шире `upvotes`. Владельцы строк могут менять поля, которые UI не показывает:

- в `projects`: как минимум `created_at` и `upvotes`;
- в `profiles`: `created_at`, а также произвольный внешний `avatar_url`;
- в `comments`: `project_id`, `created_at`, `status` и другие поля своей строки.

RLS отвечает на вопрос «какую строку», но не заменяет контроль колонок. Нужны
column grants/allowlist, отдельные RPC для бизнес-действий и защитные триггеры на
неизменяемые поля.

**Критерий закрытия:** автоматический тест перебирает все служебные колонки каждой
таблицы и подтверждает отказ обычному пользователю.

### SEC-12 — два бэкенда и миграционный drift

Production-фронтенд всё ещё использует cloud Supabase, хотя self-hosted backend уже
публичен. В cloud migration history видны только две последние миграции; более ранние
изменения применялись вручную. Cloud содержит функцию `rls_auto_enable`, которой нет
в self-hosted. Это означает, что репозиторий не воспроизводит реальное состояние БД.

**Что исправить:**

- выбрать единственный production backend;
- снять versioned schema dump с фактической БД без данных и секретов;
- привести миграции к воспроизводимой цепочке;
- сравнивать cloud/self-host/staging схемы в CI;
- после миграции закрыть старый API или ограничить его firewall/allowlist;
- отдельно проверить ротацию JWT secret и ключей при окончательном переключении.

### SEC-13 — бэкапы работают, но не зашифрованы до S3

Ежедневный backup БД, Umami и Storage выполняется успешно, bucket приватный, ротация
14 дней настроена. Это сильная сторона. Но `pg_dump` и архив Storage отправляются в
S3 без клиентского шифрования. Компрометация S3-аккаунта раскрывает базу, Auth-данные
и загруженные файлы.

**Что исправить:**

- шифровать каждый архив до загрузки (`age`, GPG или KMS envelope encryption);
- использовать отдельного least-privilege S3-пользователя только для backup bucket;
- включить versioning/immutability/Object Lock, если поддерживается;
- хранить ключ расшифровки отдельно от VPS и S3;
- провести и задокументировать квартальный restore drill.

### SEC-14 — контейнеры работают с избыточными правами

Контейнеры не privileged, внутренние сервисы не опубликованы наружу — это хорошо.
Но нет `read_only`, `cap_drop: ALL`, `no-new-privileges`; часть образов запускается
без явно заданного непривилегированного `user`.

**Что исправить:** применять эти ограничения по одному сервису на staging, добавить
`tmpfs` для нужных временных каталогов, разделить сети public/internal/data и не
разрешать frontend-proxy обращаться к сервисам, которые ему не нужны.

### SEC-15 — GitHub Actions используют изменяемые теги

Workflow применяет `actions/checkout@v4`, `configure-pages@v5`,
`upload-pages-artifact@v3`, `deploy-pages@v4` (`.github/workflows/deploy.yml:24-64`).
Разрешения job ограничены разумно, а внутренние папки корректно исключены из
артефакта. Но теги actions нужно закрепить на full commit SHA и обновлять через
Dependabot/Renovate.

### SEC-16 — существующий security-check даёт ложную уверенность

Скрипт проверяет запрет эскалации `role`, установку `is_core` и разрешённое изменение
bio. Он не проверяет `upvotes`, даты, comment fields, default privileges, функции,
Storage MIME/размер/листинг и анонимный feedback. Поэтому финальное сообщение
«ВСЁ ЧИСТО» не означает полный аудит.

Новый набор тестов должен запускаться на отдельном staging/ephemeral проекте и
покрывать матрицу `anon/member/owner/admin/service_role × CRUD × таблица/колонка`.

### SEC-17 — проверка утёкших паролей выключена

Включить Have I Been Pwned/leaked password protection в Supabase Auth и проверить
на тестовом известном скомпрометированном пароле. Не хранить и не логировать пароль
при таком тесте.

### SEC-19 — внешние изображения могут следить за посетителями

Пользователь может напрямую записать внешний `avatar_url`/image URL. Когда другой
посетитель открывает страницу, внешний сервер получает его IP, время и часть
контекста запроса. Разрешить изображения только из собственного Storage/imgproxy,
скачивать их сервером и отдавать после декодирования и очистки метаданных.

### SEC-20 — производительность RLS

Supabase Advisors показывает многократные вычисления `auth.uid()`/`is_admin()`,
несколько permissive policies и отсутствующий индекс `feedback.user_id`. Это не
обход авторизации само по себе, но под нагрузкой может превратиться в отказ в
обслуживании.

Исправить политики по рекомендациям Supabase: использовать `(select auth.uid())`,
свести перекрывающиеся политики, добавить недостающий FK-индекс и сравнить планы
запросов до/после. Руководство: [Supabase RLS](https://supabase.com/docs/guides/database/postgres/row-level-security).

### SEC-21 — npm-зависимости разработки

`npm audit` обнаружил 2 high и 3 moderate уязвимости в транзитивных dev-зависимостях.
Они не входят в статический production-артефакт. Одновременно `eslint@9` конфликтует
с peer dependency `eslint-config-standard`, ожидающим ESLint 8; автоматический
`npm audit fix --dry-run` завершается `ERESOLVE`.

Нужно согласованно обновить lint-конфигурацию и lockfile, а не запускать слепой
`npm audit fix --force`. Текущие `npm run check` и линтеры проходят.

### SEC-22 — публичность контактов должна быть осознанной

Анонимный API видит публичные профили и поля контактов; найдено три профиля. По
документам это продуктовый дизайн, а не обход RLS. Но пользователю нужно явно
объяснить, какие поля публичны, дать отдельные переключатели видимости и не
публиковать email/телефон по умолчанию. Это также должно быть согласовано с
документами о персональных данных.

## 7. Минимальный план реагирования на инцидент

Если во время запуска замечена атака:

1. Не удалять логи и не перезапускать всё без фиксации времени события.
2. Отключить затронутую функцию: signup, feedback, upload, voting или весь API.
3. Отозвать сессии и скомпрометированные ключи; `service_role`/JWT secret менять
   только по подготовленному плану, потому что это влияет на весь Auth.
4. Сохранить read-only копии логов Caddy, Auth, PostgREST, SSH и audit events.
5. Проверить целостность GitHub workflow, deploy-артефакта и БД.
6. Восстановить сервис только после закрытия причины и проверки на staging.
7. Уведомить затронутых пользователей и регулятора, если того требует характер
   утечки и применимое законодательство.

## 8. Что уже сделано правильно

- `service_role` не найден в публичном frontend/deploy artifact;
- Git-история не содержит обнаруженных PAT, private keys или service secrets;
- RLS включён на основных таблицах;
- обычный пользователь не может назначить себе admin или `is_core`;
- email confirmation включён, anonymous sign-in выключен;
- пользовательский текст в проверенных UI-компонентах выводится через безопасные
  DOM API, а внешние ссылки проверяются как HTTP(S);
- внутренние Docker-порты не опубликованы в интернет;
- UFW разрешает наружу только 22/80/443;
- TLS 1.2/1.3 и защитные заголовки на API/Stats настроены;
- рабочие бэкапы создаются ежедневно и S3 bucket не публичный;
- deploy workflow исключает `docs/`, `audits/`, `supabase/`, `infra/`, scripts и
  конфиги из GitHub Pages artifact;
- линтеры проекта проходят.

## 9. Критерий допуска к массовому запуску

Запуск допустим только когда одновременно выполнено следующее:

- все P0 и P1 закрыты и перепроверены не автором исправления;
- старый GitHub token отозван;
- накрутка `upvotes` и изменение других служебных колонок отбиваются;
- один выбранный production backend воспроизводится миграциями;
- анонимные endpoints защищены серверными rate limits/CAPTCHA;
- upload принимает только декодируемые изображения допустимого типа и размера;
- основной сайт отдаёт строгий CSP и остальные security headers;
- контейнерный scan не содержит исправляемых Critical;
- SSH effective config подтверждает отсутствие password auth;
- тест восстановления backup успешно завершён;
- мониторинг и kill switches проверены учением;
- независимый pentest не оставил открытых Critical/High.

До выполнения этих условий можно проводить только ограниченное тестирование с
доверенной группой, без публичного привлечения трафика.
