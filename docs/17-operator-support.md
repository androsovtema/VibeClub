# Операторская проверка проблем пользователей

## Быстрый вход в Supabase Studio

Из любого места в терминале выполни одну команду:

```bash
/Users/prosto/Desktop/01_Work/Products/VibeClub/infra/scripts/open-studio.sh
```

Если терминал уже открыт в корне репозитория, можно короче:

```bash
./infra/scripts/open-studio.sh
```

Скрипт сам запускает Studio на VPS, открывает защищённый SSH-туннель и страницу
`http://127.0.0.1:3000/project/default` в Google Chrome. Окно терминала должно
оставаться открытым. После работы нажми `Ctrl+C` — Studio и туннель остановятся,
основной сайт продолжит работать.

Первая загрузка новой версии Studio может занять 30–40 секунд: браузер получает
около сотни файлов через дальний SSH-туннель. После этого Chrome сохраняет
статические файлы в кэше, и повторная загрузка обычно занимает несколько секунд.
Если нужно использовать системный браузер вместо Chrome:
`WDZ_STUDIO_BROWSER=default ./infra/scripts/open-studio.sh`.

Studio не имеет публичного URL и не должен появляться в Caddy/DNS. Доступ
разрешён только владельцу SSH-ключа VPS.

## Куда смотреть при обращении пользователя

| Ситуация | Раздел Studio | Что проверить |
|---|---|---|
| Пользователь отправил сообщение «Нашли проблему?» | Table Editor → `feedback` | Время, страница, текст, статус обращения |
| Не получается войти или подтвердить почту | Authentication → Users | Существует ли аккаунт, подтверждена ли почта, когда был последний вход |
| Проблема с профилем | Table Editor → `profiles` | Роль, заполненность профиля, публичные поля |
| Не виден или не отправляется проект | Table Editor → `projects` | Автор, статус, время создания, модерация |
| Не виден комментарий | Table Editor → `comments` | Проект, автор, время и содержимое |
| Вопрос по согласию или контактам | Table Editor → `user_consents` и `profiles` | Версия, тип, `granted_at`/`revoked_at`; контакты после revoke должны быть пустыми |
| Проблема с обложкой или файлами | Storage | Есть ли объект в bucket `covers`, совпадает ли путь |

## Безопасные правила

- Сначала только смотри и фильтруй. Не редактируй строки «для проверки».
- Не меняй вручную `role`, `is_core`, `upvotes`, даты, `auth.*` или
  `user_consents`: для этого существуют защищённые UI/RPC-потоки.
- Не удаляй пользователя из Authentication: удаление не гарантирует мгновенный
  отзыв уже выданного access token и может оставить связанные данные.
- Не копируй в чат email, ФИО, UUID, JWT, телефоны и контакты. Для совместной
  диагностики достаточно страницы, времени, последовательности действий и
  агрегатов без идентификаторов.
- SQL Editor используй только для `SELECT`, пока запрос отдельно не проверен и
  изменение production явно не согласовано.

## Чего Studio не показывает

Studio помогает с БД, Auth и Storage, но не заменяет диагностику браузера и
контейнеров. Если данные выглядят правильно, а ошибка повторяется, запиши:

1. страницу и примерное время;
2. что пользователь нажал и что ожидал;
3. точный текст ошибки;
4. воспроизводится ли проблема у тебя.

После этого передай описание Codex: он сопоставит время с browser console,
Auth/PostgREST/Storage и серверными логами без публикации секретов.
