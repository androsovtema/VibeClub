## 1. Read-only baseline

- Репозиторий: `main`, `HEAD = origin/main = bd68c13`.
- Public origin: `https://wedesignerz.com` отвечает через GitHub Pages/Fastly
  (`Server: GitHub.com`, `Via: varnish`, адреса GitHub Pages).
- Индексация закрыта: живой `robots.txt` содержит `Disallow: /`.
- Production backend: self-hosted Supabase на `api.wedesignerz.com`.
- Production analytics: self-hosted Umami на `stats.wedesignerz.com`.
- Production почта: внутренний `mail-bridge` отправляет через Web API
  Unisender Go.
- Резервные копии: приватный S3-совместимый bucket Timeweb; ежедневные
  дампы основной БД и Umami плюс архив Storage, ротация старше 14 дней.
- Supabase Studio не имеет публичного URL: доступ только через SSH-туннель
  владельца ключа.
- Агрегатная read-only проверка production 2026-07-17 подтвердила, что самая
  ранняя текущая запись `auth.users`/`profiles` создана 2026-07-07. Поэтому
  планируемая дата публичного сбора 2026-07-18 не может использоваться как
  дата начала обработки: технически обработка уже началась не позднее
  2026-07-07.
- Изменений DNS, БД, production-конфигурации и регистраций в ходе preflight

После read-only проверки доступ был организован через российский VPS

## 2. Категории субъектов

| Категория | Фактическое участие |
|---|---|
| Посетители сайта | Получение страниц и assets, технические сетевые данные, обезличенная аналитика |
| Зарегистрированные участники | Auth-аккаунт, профиль, согласия, сессии |
| Авторы проектов | Профиль, проектный контент, файлы, статусы модерации |
| Авторы комментариев и реакций | Комментарии, категория комментария, связь с проектом, лайки |
| Отправители обращений | Страница, текст обращения, необязательный контакт, связь с аккаунтом при наличии сессии |

Сервис не запрашивает паспортные, платёжные, медицинские данные и отдельный
признак возраста. Он не позиционируется как сервис для детей; отдельный
возрастной контроль сейчас отсутствует.

## 3. Категории данных

### Непубличные данные

- email аккаунта, парольный хэш и технические Auth/session metadata;
- закрытый журнал согласий: тип, версия политики, время выдачи/отзыва, scope;
- ФИО и подтверждённый контакт только как доказательство отдельного согласия
  на распространение;
- обращения в поддержку, включая необязательный контакт;
- технические сетевые данные: IP-адрес, user agent, referrer, timestamps и
  сведения о событиях безопасности в объёме, который фактически получают
  origin, API, CAPTCHA, аналитика и инфраструктурные журналы.

### Публичные данные и пользовательский контент

- имя или псевдоним, аватар, bio, навыки и отметки `open_to`;
- проекты: название, описание, обложка/галерея, ссылка, теги, инструменты,
  стадия и `looking_for`;
- опубликованные комментарии и реакции;
- только после отдельного активного согласия на распространение: Telegram,
  сайт, GitHub, телефон, публичная почта, название и URL своей ссылки.

Почта входа не становится публичной автоматически. ФИО из закрытого журнала
согласий не входит в публичный профиль.

## 4. Цели обработки

1. Регистрация, идентификация, вход, восстановление доступа и управление
   пользовательской сессией.
2. Ведение публичных профилей участников клуба.
3. Приём, модерация и публикация проектов и медиафайлов.
4. Комментарии, реакции и взаимодействие участников вокруг проектов.
5. Обратная связь, поддержка, обработка жалоб и модерация.
6. Фиксация, исполнение и отзыв согласий, включая отдельное согласие на
   распространение контактных данных.
7. Работа, безопасность, резервное копирование, восстановление и мониторинг
   сервиса.
8. Обезличенная продуктовая аналитика без cookies и без привязки события к
   публичному профилю.
9. Доставка служебных писем Auth: подтверждение, вход и восстановление.

Монетизация, реклама, скоринг и персонализированные рекомендации в текущую
цель обработки не входят.

## 5. Операции и способ обработки

Фактически выполняются: сбор, запись, систематизация, накопление, хранение,
уточнение, извлечение, использование, предоставление доступа, распространение
только по отдельному согласию, блокирование, отзыв, удаление и уничтожение.
Для аналитики также применяется обезличивание/агрегация.

Основной процесс автоматизирован и идёт с передачей по Интернету. При этом
оператор вручную модерирует проекты и обращения через web-интерфейс. В live
форме доступны варианты «автоматизированная», «неавтоматизированная» и
«смешанная», отдельно — наличие передачи по внутренней сети юридического лица
и по сети Интернет. Технической картине соответствует «смешанная», «без
передачи по внутренней сети юридического лица», «с передачей по сети
Интернет». Окончательный выбор остаётся правовой квалификацией оператора.

## 6. Системы и получатели

| Контур | Роль | Размещение / статус | Что важно для формы |
|---|---|---|---|
| `wedesignerz.com` | Статический сайт | GitHub Pages/Fastly до T-FRONT-VPS | Иностранный origin получает сетевые идентификаторы посетителя; пользовательские формы пишут не в Pages, а напрямую в RU backend |
| Self-host Supabase | Auth, Postgres, PostgREST, Storage | RU-VPS Timeweb | Primary database и Storage; точные реквизиты юрлица и адрес ЦОД взять из договора/кабинета |
| `mail-bridge` + Unisender Go | Служебные Auth-письма | bridge на RU-VPS; внешний Web API | Проверить договорного получателя, состав передаваемых данных, страну и сроки хранения |
| Self-host Umami | Аналитика | RU-VPS Timeweb | По текущей публичной политике — без cookies и без хранения raw IP; фактическую настройку и точный срок хранения аналитики нужно подтвердить |
| Timeweb S3 | Backup | Приватный bucket | DB + Umami + Storage, текущая ротация 14 дней; клиентское шифрование backup не настроено |
| Supabase Studio | Операторский доступ | Только SSH-туннель | Не публичный; доступ владельца SSH-ключа, сначала read-only |
| Google Fonts | Шрифты | Внешние browser requests | Автоматический запрос браузера к `fonts.googleapis.com` и `fonts.gstatic.com` |
| Cloudflare Turnstile | CAPTCHA | Внешние browser/server requests | Загружается в Auth-сценариях; состав technical data и получателя сверить по актуальным условиям |
| Внешние `avatar_url` | Изображение профиля | Потенциально любой HTTPS-host | Код подставляет URL в `<img>`, а CSP разрешает `img-src https:`; даже при отсутствии такого запроса на проверенной главной возможность остаётся |

Ссылки на Telegram, YouTube, GitHub и пользовательские сайты сами по себе не
создают автоматический запрос при загрузке страницы: получатель появляется
после осознанного клика посетителя. Это нужно отличать от Google Fonts,
Turnstile и внешнего `avatar_url`.

Старый Supabase Cloud host всё ещё присутствует в CSP как rollback-allowlist,
но production-клиент использует `api.wedesignerz.com`; на проверенной главной
живых запросов к старому cloud host не наблюдалось. `supabase-js` хранится в
репозитории, поэтому `esm.sh` не является действующим runtime-получателем.

## 7. Фактические меры защиты

- TLS для публичных сервисов; HTTP перенаправляется на HTTPS.
- RLS и server-side ownership checks для exposed-таблиц и Storage.
- Публичный frontend использует только anon key; service-role не публикуется.
- CAPTCHA на регистрации, минимальная длина пароля 12 символов, подтверждение
  email и отзыв тестовых сессий после проверок.
- Только Caddy публикует порты наружу; БД и внутренние сервисы напрямую не
  экспонируются.
- SSH-доступ по ключу; парольный SSH выключен.
- Studio запускается по требованию и доступен только через localhost-туннель.
- Приватный backup bucket, ежедневный backup, restore-test, watchdog и внешние
  health-мониторы.
- Закрытый журнал согласий и отдельные RPC grant/revoke; при revoke публичные
  контакты очищаются, а история согласия сохраняет `revoked_at`.
- CSP ограничивает основные script/connect/frame origin, но текущий
  `img-src https:` остаётся широким исключением для внешних изображений.

Остаточные технические риски, которые нельзя превращать в формулировку
«полностью защищено»: backup без клиентского шифрования, широкий `img-src
https:`, незафиксированные здесь сроки части журналов/аналитики и иностранные
browser-зависимости.

На VPS не обнаружены установленные сертифицированные СКЗИ или продукты с
классом КС1/КС2/КС3/КВ/КА. Фактически используются стандартные Caddy TLS,
OpenSSH и OpenSSL; у них нет требуемых формой класса, изготовителя и серийного
номера российского СКЗИ. Поэтому ветка «криптографические средства
используются» для текущей конфигурации не подтверждена: в уведомлении следует
выбрать «не используются», а TLS/SSH продолжать указывать среди технических
мер защиты.

## 8. Хранение, отзыв и удаление

| Данные | Текущее правило |
|---|---|
| Аккаунт, профиль и контент | До запроса на удаление/прекращения цели обработки; числовой срок в политике не установлен |
| Публичные контакты | До отзыва отдельного согласия; при revoke поля очищаются сразу, запись согласия получает `revoked_at` |
| Закрытый журнал согласий | История сохраняется после revoke; при удалении Auth-пользователя действует cascade. Требуемый юридический срок хранения доказательства нужно подтвердить отдельно |
| Feedback | До обработки обращения и далее по пока не утверждённому сроку; нужен явный retention rule |
| Production backup | Текущая техническая ротация — 14 дней |
| Umami и серверные журналы | Точные retention settings нужно снять перед финальной отправкой |
| Mail provider | Срок хранения и удаление сверить по договору/кабинету Unisender Go |

Публичная политика обещает удаление «в разумный срок», но конкретный срок не
зафиксирован. Перед подачей нельзя выдумывать число: нужно либо принять
операторское правило, либо честно указать условие прекращения обработки,
которое допускает актуальная форма.

## 9. Трансграничная инвентаризация

### До T-FRONT-VPS

- GitHub Pages/Fastly — автоматическая доставка всех страниц и assets;
- Google Fonts — автоматическая загрузка CSS и шрифтов;
- Cloudflare Turnstile — Auth/CAPTCHA-сценарии;
- Unisender Go — служебные письма, если договорный контур или обработка
  включает иностранного получателя;
- произвольный HTTPS-host внешнего аватара — потенциально при наличии такого
  URL в публичном профиле.

### После T-FRONT-VPS

GitHub Pages/Fastly должны уйти из активного origin после 72-часового окна
наблюдения. Google Fonts, Cloudflare Turnstile, договорный контур почты и
потенциальные внешние аватары останутся, пока их отдельно не устранить или не
ограничить.

**Технический вывод:** текущая архитектура не даёт основания отметить вопрос
трансграничной передачи как автоматически закрытый. Для отдельного
уведомления нужно подтвердить юридические лица-получатели, страны, состав
передаваемых данных, цели и меры защиты по официальной форме и актуальным
договорам. Старые Supabase Cloud, Resend и Umami Cloud не являются текущими
production-получателями и не должны копироваться в форму.

В основной live-форме подтверждён обязательный выбор «осуществляется» или «не
осуществляется». По части 3 статьи 12 152-ФЗ уведомление о намерении
осуществлять трансграничную передачу направляется отдельно от уведомления об
обработке по статье 22. Поэтому отмечать отсутствие передачи только ради
отправки основной формы нельзя, а отправлять основной черновик с наличием
передачи до завершения отдельной инвентаризации — текущий стоп-фактор.

## 10. Что Тёма подтверждает приватно

Значения вводятся только в официальную форму и не пересылаются в чат:

- статус и реквизиты оператора, данные ЕСИА, ИНН;
- регистрационный адрес/регион, телефон и контактный email;
- точная дата начала обработки;
- основание и условие прекращения обработки;
- реквизиты Timeweb и точный адрес российского ЦОД из договора/кабинета;
- отдельная площадка хранения S3 backup;
- договорные реквизиты и фактический контур Unisender Go;
- юридические лица, страны и меры по каждому подтверждённому случаю
  трансграничной передачи;
- выбранная квалификация способа обработки;
- сроки Feedback, Umami, серверных журналов и доказательств согласия;
- сведения об ответственном за организацию обработки, с пониманием того, что
  форма предупреждает об их общедоступности.

После отправки идентификатор, ключ записи и PDF/подтверждение сохраняются в
приватном хранилище владельца. В Git попадают только дата и нейтральный статус.

## 11. Стоп-условия

Этап нельзя считать закрытым, пока одновременно не выполнено следующее:

1. Тёма подтвердил дату начала обработки и частные реквизиты.
2. Проверены реквизиты российского ЦОД и договорный почтовый контур.
3. Принято операторское retention rule для незакрытых категорий.
4. По официальной форме и, по возможности, с профильным специалистом принято
   решение о составе отдельного уведомления по трансграничной передаче.
5. Основное уведомление отправлено Тёмой через официальный канал, а
   подтверждение сохранено вне Git.

До подтверждённой подачи не обновлять launch-документы статусом «подано», не
готовить финальный T-FRONT-VPS handoff и не начинать DNS/VPS-cutover.
