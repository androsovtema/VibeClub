# 04 — Задачи реализации (Sonnet / Terra)

Документ для моделей-исполнителей. Sonnet ведёт продуктовые задачи, Terra —
infra/cutover по отдельным промптам в `docs/prompts/terra/`. Актуальная очередь
находится в конце файла; исторические задачи выше сохраняются как журнал.
Контекст — в `00..03`, `05` и `08`. Визуальный стиль и шрифты сохраняем:
Onest + JetBrains Mono.

Общие правила:
- Стек: статический HTML/CSS/JS, без фреймворка и сборки. Сейчас origin —
  GitHub Pages; целевой origin до анонса — Caddy на RU-VPS (T-FRONT-VPS).
- UI-строки — в `js/i18n/ru.js` (объект-словарь), не хардкодить вразнобой в HTML.
- Supabase-ключи (anon) — в `js/config.js`. `service_role` нигде не использовать.
- Безопасность данных = RLS. Любая таблица без включённого RLS = баг.
- Все запросы к Supabase — через `@supabase/supabase-js`, вендоренный локально
  в `js/vendor/` (CDN-импорт убран, см. `16-security-status.md`, SEC-07).

## T0 — Подготовка проекта Supabase (ручной шаг + SQL)
**Делает:** Тёма создаёт проект на supabase.com (free). Sonnet готовит SQL-миграцию.
**Выход:**
- `supabase/schema.sql` — создание таблиц `profiles`, `projects`, `comments`,
  `project_upvotes` по схеме из `01-architecture.md`.
- Триггер: при регистрации в `auth.users` создаётся строка `profiles`.
- Триггер: пересчёт `projects.upvotes` при изменении `project_upvotes`.
- Включён RLS на всех таблицах + все политики из `01-architecture.md`.
- Bucket Storage `covers` (публичное чтение, запись для аутентифицированных).
**Критерии приёмки:**
- SQL применяется без ошибок.
- Анонимный клиент НЕ может вставить `projects` со `status='published'`.
- Анонимный клиент НЕ может писать в чужой профиль.
- Залогиненный может вставить проект только с `author_id = своему uid` и `status='pending'`.

## T-D — Новый визуальный стиль (выполняется параллельно T1–T4, ДО T5)
**Контекст:** `06-design-direction.md`. Источник стиля — дизайн-система из Claude Design
(через `/design-sync`). Это не косметика, а смена сути визуала: уход от элегантной
«природной» эстетики с засечками к энергии кода/ИИ/созидания. Сайт должен сам выглядеть как
крутой навайбкоженный продукт.
**Выход:**
- `styles.css` (или новый `css/tokens.css`) — токены из дизайн-системы как CSS-переменные
  в `:root`: цвета, градиенты, типографика (дисплей + текст + моно), spacing, радиусы,
  тайминги анимаций.
- Подключение новых шрифтов (геометрический гротеск + читаемый сан-сериф + моно). Cormorant
  Garamond убрать.
- Базовые компоненты по `06`: кнопки (главный CTA «Вступить» — самый заметный), карточка
  проекта, чипы тегов/инструментов, поля форм, хедер/навигация, бейдж «флагман We Designerz»,
  блок сообщения, пустые состояния, тосты.
- Анимации на CSS/малом JS (без тяжёлых либ): hero, hover карточек, печать текста, переходы.
  Уважать `prefers-reduced-motion`.
**Критерии приёмки:**
- Старая эстетика (Cormorant, природно-студийный вид) полностью убрана.
- Контраст текста проходит WCAG AA.
- Анимации не тормозят, отключаются при `prefers-reduced-motion`.
- Стиль — единый источник правды через CSS-переменные; страницы T5/T6 строятся на нём.
- На hero возникает эффект «вау, это само навайбкожено».
- Статус первого прохода: базовые токены/шрифты/hero-терминал СДЕЛАНЫ. Осталась
  де-шаблонизация — см. T-D2.

## T-P3 — Никнеймы: колонка `username`, правка в кабинете, чтение по `?u=` (P1)

красивый `/@ник` в адресной строке — отдельная T-P4 поверх этого.

- **Миграция** (`supabase/migrations/`): `alter table public.profiles add column username text unique;`

  колонке. Идемпотентно. `username` остаётся nullable — существующие профили без ника продолжают
  жить по `?id=`.

  источник истины — unique constraint. Дополнительно на уровне БД: триггер нормализации
  `lower(trim(username))` перед insert/update (клиентский lower — не гарантия: прямой вызов

## T-P4 — Красивый URL профиля `/@ник` через `404.html`-фолбэк (P1, зависит от T-P3)

**Зачем:** `profile.html?u=tema` работает, но некрасив. Хотим как YouTube — `wedesignerz.com/@tema`.
GitHub Pages — чистая статика, серверных rewrite нет, поэтому единственный механизм это фолбэк:
GH отдаёт `404.html` на любой несуществующий путь, а JS в нём разбирает `location.pathname`.
Выбран префикс `@` (не голый `/ник`): исключает коллизии с реальными страницами и читается как хэндл.

- **`404.html`:** скрипт читает `location.pathname`. Если совпадает с `^/@([a-z0-9_]{3,20})$` —
  это профиль: грузить по `username` (переиспользовать рендер из `js/profile.js`, вынести общую
  логику в модуль, не дублировать) и показать карточку, оставив URL `/@ник` в строке
  (`history.replaceState` не нужен — путь уже такой). Иначе — обычный экран 404.
- **Резервный список.** `@about`, `@projects`, `@submit`, `@me`, `@profile`, `@admin`, `@index`,
  `@project`, `@404` и прочие имена страниц — запретить в T-P3 при выборе ника (дополнить валидацию),
  чтобы хэндл не «затмевал» будущую страницу.
- **Исходящие ссылки.** Там, где T-P3 ставит `?u=<ник>`, для профилей с ником отдавать красивый
  `/@<ник>`; `?id=`/`?u=` оставить рабочими (входящие старые ссылки, боты).
- **SEO-компромисс — задокументировать, не игнорировать.** Контент под `/@ник` отдаётся с HTTP-статусом
  404 (природа фолбэка). Пользователь видит страницу нормально, но поисковик видит 404 → слабая
  индексация профилей. На старте приемлемо (профили некритичны для SEO). В приёмке зафиксировать
  как известное ограничение; при необходимости позже — прероллинг/сайтмап профилей.

**Приёмка:** `wedesignerz.com/@tema` показывает
профиль, URL в строке остаётся красивым; несуществующий `@ник` и обычный битый путь дают нормальный
404; реальные страницы (`/about`, `/projects.html` и т.д.) не перехватываются; старые `?id=`/`?u=`
работают. Ограничение по HTTP-404-статусу отмечено в статусе задачи.

**Статус (2026-07-10, по вердикту аудита):** приоритет понижен. Не рекламировать
`/@ник` как SEO-профиль на GitHub Pages (поисковик видит HTTP 404). Канонический
публичный URL до смены хостинга — `profile.html?u=<ник>`. Возвращаться к T-P4 после
хостинга с rewrite/edge или статического prerender профилей.

## T18 — Антиспам и серверные лимиты (P0, до анонса; аудит 2026-07-10)

**Зачем:** RLS защищает права на строки, но не сдерживает злоупотребление. Сейчас все
`check` в `supabase/schema.sql` — только enum/массивы; ни одного ограничения длины на
text-полях, rate limit отсутствует. Авторизованный пользователь может слать мегабайтные
комментарии в цикле. Вердикт: `audits/current-review/2026-07-10-verdict.md`.

**Выход (миграция в `supabase/migrations/`, идемпотентная, + синхронизация `schema.sql`):**
- Лимиты длины на уровне БД: `char_length` check на все пользовательские text-поля —
  `profiles` (display_name, bio, ссылки), `projects` (title, description, ссылки),
  `comments` (body). Разумные границы (например: title ≤ 120, body комментария ≤ 2000,
  bio ≤ 500 — финальные числа согласовать с фактическим UI).
- Лимит количества элементов массивов (`tags`, `tools`, `looking_for`, `skills`,
  `open_to`) — по образцу существующего `projects_images_max`.
- Cooldown **только на комментарии**: не чаще 1 комментария в 20 сек и N в час
  на пользователя. Защита должна быть серверной: БД-триггер **либо** insert через
  RPC/Edge Function с закрытым прямым `INSERT`; одного клиентского таймера недостаточно.
  Клиентская ошибка показывается человеко-читаемо, без падения в консоль.
- Апвоуты в первом проходе не throttle'им: составной PK уже запрещает повторный
  голос за один проект, а cooldown ухудшит обычное голосование и не остановит
  мультиаккаунты. Вернуться к этому только при живом сигнале злоупотребления.
- Подтверждённая почта — через конфигурацию Supabase Auth: перед продом включить
  **Confirm Email** и выключить тестовый **Auto Confirm**. Неподтверждённый
  пользователь не должен получать сессию, следовательно не дойдёт до insert в
  `projects`/`comments`. Не проверять `email_confirmed_at` через `auth.jwt()` в RLS:
  этот claim не гарантирован в JWT. Проверить настройку реальным e2e-прогоном.
- Клиент: `maxlength` на соответствующих полях форм + инлайн-ошибки (лимиты клиента
  = лимиты БД, единый источник — задокументировать соответствие в комменте миграции).

**Приёмка:** превышение длины отбивается и БД (insert из консоли под обычной учёткой),
и формой; серия быстрых комментариев упирается в серверный cooldown с внятной ошибкой;
при выключенном Auto Confirm новая учётка не получает сессию до подтверждения письма и

## T-CONSENT — Журналирование согласия на обработку ПДн (P0-legal; внешний аудит 2026-07-14)

**Зачем:** чекбокс согласия при регистрации (T20) только разблокирует кнопку —
факт, дата и версия политики нигде не сохраняются (`js/ui/authModal.js`,
`js/auth.js`). При споре/проверке доказать согласие нечем. Плюс T23 сделал
телефон/почту/контакты публичными — для **распространения** ПДн по 152-ФЗ
ст. 10.1 нужно отдельное согласие, его нет.

**Принятое техническое решение:** отдельный приватный журнал
`public.user_consents`, а не изменяемые поля профиля. Минимальные поля:
`id`, `user_id`, `consent_type` (`processing`/`dissemination`),
`policy_version`, `granted_at` (серверное время), `revoked_at`, `scope` JSON;
для `dissemination` дополнительно сохраняются непубличные
`subject_full_name` и `subject_contact`, требуемые приказом № 18.
Пользователь читает только свои записи; `anon` не читает; прямое изменение
`granted_at`/`user_id` запрещено. Актуальную версию политики держать одной
константой фронта и показывать рядом со ссылкой на `privacy.html`.

**Регистрация:** фронт передаёт только версию согласия в signup metadata;
триггер создания пользователя сам фиксирует серверное время и тип
`processing`. Клиентское `consent_at` не принимать как доказательство. Если
записи в журнале нет, регистрация не должна считаться полностью принятой.
`raw_user_meta_data` пользователь может менять сам: сервер обязан принять
signup только при точном совпадении значения с разрешённой версией и после
записи журнала больше не использовать metadata для авторизации.

**Публичные контакты:** в `/me` отдельный, по умолчанию пустой чекбокс
«Разрешаю публиковать указанные контакты» со ссылкой на текст согласия. Сохранение
непустых `phone`, `email_public`, `telegram`, `github`, `website` и custom-link
без активного `dissemination`-согласия блокируется. Отзыв согласия атомарно
ставит `revoked_at` и очищает/скрывает публичные контакты. Существующие контакты
трёх профилей нельзя молча считать согласованными: перед анонсом владельцы
должны подтвердить их заново; безопасный дефолт миграции — не публиковать до
подтверждения. Запрет должен стоять на уровне БД и отбивать прямой REST PATCH,
а не только выключать кнопку во фронте.
ФИО для согласия вводится отдельно от публичного имени/ника, сопровождается
явным пояснением «не публикуется», а контакт берётся сервером из почты аккаунта;
клиент не может подменить `subject_contact` в RPC.

действие атомарно сохраняет active processing v4 с серверной датой и не теряет
историю v2; повторный вызов идемпотентен; RLS не раскрывает журнал; есть

## T-CONSENT-UX — Понятное объяснение отдельного согласия (P0-legal UX; фидбэк Тёмы 2026-07-15)

запросить полное имя и показать отдельный чекбокс. Сам блок «Контакты» идёт
ниже и говорит только о заполняемых публичных полях.

- заголовок: «Публикация контактов»;
- вводный текст: «Контакты увидят все посетители профиля. Поэтому по требованиям
  закона о персональных данных нам нужно отдельно получить и сохранить твоё
  согласие.»;
- label поля: «Как тебя зовут полностью»;
- placeholder: «Фамилия Имя Отчество (если есть)»;

  публичного профиля.»;
- ссылка: «Что именно публикуется и как отозвать согласие»;
- hint блока «Контакты»: «Все поля необязательны. После согласия выше
  заполненные контакты будут видны всем на странице профиля.»

**Юридическая граница:** ФИО и подтверждённая почта остаются только в закрытом

## T-CONSENT-AUTH — Разделить согласие на ПДн и правила клуба (P0-legal; ревью 2026-07-16)

**Проблема:** регистрация сейчас использует один чекбокс с формулировкой «Даю
согласие на обработку моих персональных данных … и принимаю правила клуба».
С 1 сентября 2025 года часть 1 статьи 9 152-ФЗ требует оформлять согласие на
обработку ПДн отдельно от иной информации и документов, которые подтверждает
субъект. Совмещать согласие на ПДн и принятие правил в одном контроле перед
live-применением нельзя.

**Принятое решение:** в signup-форме два самостоятельных, по умолчанию пустых
чекбокса:

1. «Даю согласие на обработку моих персональных данных на условиях политики
   конфиденциальности.» — отдельная ссылка только на `privacy.html`;
2. «Принимаю правила клуба.» — отдельная ссылка только на
   `privacy.html#rules`.

Кнопка «Вступить в клуб» доступна только при обоих подтверждениях. Submit
дополнительно проверяет оба значения, чтобы нельзя было обойти гейт Enter/
программным submit. В серверный signup metadata по-прежнему передаётся только
версия processing consent; принятие правил не смешивается с
`public.user_consents` и не расширяет DB-схему этой задачи.

**Версия:** из-за изменения текста processing consent до live версия
повышается `privacy-2026-07-15-v3` → `privacy-2026-07-16-v4` одновременно в
`js/consent.js`, ещё не применённой migration и `supabase/schema.sql`.

**Приёмка:** два независимых label/checkbox, оба не отмечены по умолчанию;
правильные ссылки и focus; кнопка активна только при обоих; снятие любого снова
блокирует кнопку; submit-handler явно отбивает отсутствие каждого согласия;
логика входа/recovery не затронута; desktop/375 px без overflow; `npm run
check`, `git diff --check`, `node --check js/ui/authModal.js` зелёные. Никаких
live migration, VPS, commit, push или deploy.

**Статус:** ВЫПОЛНЕНО SONNET И РЕВЬЮ ПРИНЯТО 2026-07-16. Два независимых
checkbox/label и submit-guards реализованы, версия синхронно поднята до
`privacy-2026-07-16-v4`; `js/auth.js`, metadata и DB-контракт не менялись.

## T-AUTH-UX2 — Блокеры финальной production-приёмки Auth/UX (P0; фикс-лист 2026-07-16)

**Проблема:** финальная приёмка нашла 4 дефекта в модалке логина/регистрации и
`/me`, из-за которых `T-CONSENT` нельзя было принять:

1. На 375 px placeholder непубличного ФИО в `/me` («Фамилия Имя Отчество (если

2. Ни в одном password-поле не было доступного «показать/скрыть пароль».
3. Email и пароль в Auth принимали кириллицу без ранней валидации — уходили в

## T-FRONT-VPS — Перенос статического фронта с GitHub Pages на RU-VPS (P0 до анонса)

**Зачем:** в принятом порядке Claude Fable после локализации БД и уведомления

`wedesignerz.com` всё ещё обслуживается GitHub Pages/Fastly: это иностранная
зависимость, а Pages не даёт управлять полным набором HTTP response headers.

**Выход:** тот же статический HTML/CSS/JS без фреймворка и сборки отдаёт Caddy
на текущем RU-VPS. GitHub остаётся репозиторием, но не публичным origin сайта.

**Что подготовить отдельным промптом Terra после T-CONSENT и уведомления РКН:**

- Caddy-блок для `wedesignerz.com` (+ явное решение для `www`) с `file_server`,
  gzip/zstd, HTTPS, 404 и response headers: HSTS, nosniff, frame deny,
  Referrer-Policy, Permissions-Policy и CSP. Meta-CSP пока оставить как
  defense-in-depth; значения header/meta не должны расходиться.
- Сборку `_site/` с тем же allowlist/исключениями, что сейчас в

- Атомарный deploy в `/srv/wedesignerz/releases/<id>` → symlink `current`,
  отдельный непривилегированный deploy-user, без SSH root из GitHub Actions.

- Cutover DNS с пониженным TTL, 72-часовой rollback-период на Pages и только

## T-IMG — Оптимизация статичных картинок студии (P2; запрос Тёмы 2026-07-14)

**Зачем:** страницы «О нас» и кейсы студии (`about.html`, `projects/*.html`) грузят
сырые JPEG — `assets/images/projects/` весит ~11 МБ, плюс `carousel/` 1 МБ и
`team/` 0.5 МБ. Ни конвертации в современный формат, ни responsive-размеров:
на медленной сети картинки видимо «доезжают».

**Что сделать:**
1. Конвертировать JPEG/PNG в **WebP** (качество ~80, обычно −60…70% веса).
   Инструмент — `cwebp` (brew) или скрипт; Тёма может прогнать через внешний
   сервис (Squoosh/TinyPNG) — результат тот же, класть рядом с оригиналами.
2. В HTML — `<picture>` с WebP + JPEG-фолбэком, либо просто заменить `src`
   на `.webp` (поддержка WebP везде с 2020).
3. `loading="lazy"` + явные `width`/`height` (убрать layout shift) на всех
   картинках ниже первого экрана.
4. Крупным обложкам кейсов — уменьшенные версии под реальный размер контейнера
   (сейчас грузится оригинал в полном разрешении).
5. `og-image.png` (248 КБ) не трогать — соцсети WebP в OG понимают плохо.

**Приёмка:** суммарный вес `assets/images/` ≤ 4 МБ; Lighthouse на `about.html` —
без варнингов «serve images in next-gen formats» и «properly size images»;
визуально картинки не деградировали.
