# 01 — Архитектура и данные

## Стек

- **Фронт:** оставляем статический HTML/CSS/JS, без фреймворка и сборки.
  **Сейчас** хостинг — GitHub Pages, домен `wedesignerz.com`, деплой через
  `.github/workflows/deploy.yml` (push в `main` → публикация). Workflow собирает
  `_site/` только из файлов сайта: `docs/`, `audits/`, `supabase/`, `CLAUDE.md` и
  конфиги на публичный сайт **не попадают** (правило от 2026-07-06 — до этого весь
  репозиторий, включая внутренние доки, был публично доступен по прямым URL).
  **Целевая схема до анонса:** тот же статический фронт отдаёт Caddy на RU-VPS
  (T-FRONT-VPS), с response headers и атомарным deploy/rollback; GitHub остаётся
  репозиторием, но не origin.
- **Бэкенд:** API-совместимый Supabase. До T-CUTOVER фронт смотрит в Supabase
  Cloud; целевая схема — self-hosted Postgres + GoTrue + PostgREST + Storage на
  RU-VPS. Клиент `@supabase/supabase-js` вендорен локально в `js/vendor/`.
- **Чат сообщества:** внешний Telegram-чат — дополнительный канал общения.
  «Вступить» открывает auth/членство на сайте, не ведёт прямо в Telegram.

Почему так: рабочую статику, API-контракт и визуал не переписываем; меняются
только origin фронта и размещение совместимого бэкенда. Это снижает миграционный
риск и сохраняет существующий клиент/RLS.

## Ключевой принцип безопасности: anon key публичный → защита через RLS

В статическом сайте `anon`-ключ Supabase виден в исходниках браузера. Это нормально и
ожидаемо — но значит, что **вся защита данных строится на Row Level Security (RLS)** в
Postgres, а не на «спрятанном» ключе. Каждая таблица обязана иметь включённый RLS и явные
политики. Без этого любой сможет писать в БД напрямую. Это самый важный пункт всей реализации.

`service_role`-ключ **никогда** не попадает на фронт. Он только для серверных задач
(если появятся — например, Edge Functions для модерации).

## Авторизация

Аудитория в основном РФ, где **Google-авторизация заблокирована** — поэтому Google не может
быть единственным/обязательным входом. Минимизируем барьер, но без внешних зависимостей:

**MVP:**
- **Почта + пароль** — основной способ. Работает везде, без внешних сервисов.
- **Email magic link** — беспарольный запасной.

**Бэклог (после запуска):**
- **Telegram-логин** — сильный кандидат: сообщество живёт в Telegram, вход одним тапом.
  Не нативный провайдер Supabase — нужна проверка подписи Telegram через Edge Function.
- **Телефон + SMS-код** — требует платного SMS-провайдера (Twilio и т.п.) и настройки.
- Прочие OAuth (Google и др.) — опционально доп. кнопкой, но не как основной путь.

GitHub OAuth не делаем — барьер для целевой аудитории (не-технари).

Добавить проект и писать сообщения можно только залогинившись. Просмотр — без логина.

## Политики RLS (по таблицам)

**`profiles`**
- SELECT: всем (публичные профили).
- INSERT/UPDATE: только владелец (`auth.uid() = id`).

**`projects`**
- SELECT: всем, где `status = 'published'`. Автор видит свои в любом статусе. Админ видит все.
- INSERT: аутентифицированным, только `author_id = auth.uid()` и `status = 'pending'`
  (нельзя самопубликоваться).
- UPDATE статуса (`published`/`rejected`) и `is_core`: только `role = 'admin'`.
- UPDATE контента своей записи (пока `pending`): автор.
- DELETE: автор своей записи или админ.

**`comments`**
- SELECT: всем, где `status = 'published'` и родительский проект опубликован.
- INSERT: аутентифицированным, `author_id = auth.uid()`.
- UPDATE/DELETE: автор своей записи или админ (скрытие = `status='hidden'`).

**`project_upvotes`**
- SELECT: всем. INSERT/DELETE: только `user_id = auth.uid()`.

## Модерация и анти-спам

- Проекты публикуются после ручного одобрения админом (`status pending → published`).
  Простая админ-вьюшка — отдельная страница `/admin`, видна только роли `admin`.
- Комментарии — автопубликация, но с правом скрыть. На будущее — рейт-лимит.
- Защита форм: hCaptcha/Turnstile на сабмите проекта (этап после MVP, если пойдёт спам).
- Storage: лимит размера обложки, разрешённые типы (jpeg/png/webp).

## i18n-готовность (EN позже, не сейчас)

Не строим английскую версию на запуск, но не закрываем дверь:
- Все UI-тексты держим в одном месте — `js/i18n/ru.js` (объект-словарь `{ key: text }`),
  а не разбросаны по HTML вперемешку. Позже добавляется `en.js` и переключатель.
- Пользовательский контент (проекты/комменты) остаётся на языке автора — не переводим.
- URL-структура закладывается под префикс языка на будущее (`/`, позже `/en/`), но сейчас
  без префикса.

Цель этапа MVP — чтобы добавление EN потом было «добавить файл словаря + переключатель»,
а не «переписать весь сайт».

## Конфиги и секреты

- Публичные `SUPABASE_URL` и `SUPABASE_ANON_KEY` лежат в `js/config.js` (это нормально,
  они и так публичны — защита в RLS).
- Никаких `service_role` ключей в репозитории.
- `.env` не нужен для статики; если появятся Edge Functions — секреты в настройках Supabase.
