# 12 — Ранбук: когда что прогонять

Проект оброс проверками. Этот файл — единственное, что нужно держать под рукой:
что запускать, когда и зачем. Разложено по «моментам», а не по инструментам —
смотри свой момент и делай по списку.

## Быстрая шпаргалка

| Момент | Команда / действие | Зачем |
|---|---|---|
| После правок кода (Sonnet/Terra отчитался) | `npm run check` | линтеры и repo-checks = 0 |
| Перед коммитом | `npm run check` | не коммитить сломанное |
| После применения RLS-миграции | `WDZ_TEST_JWT='<access_token>' npm run security-check` | привилегии не утекают при включённой CAPTCHA |
| Перед анонсом | весь чек-лист `docs/05-launch.md` | ничего не забыто |
| Раз в месяц / после Supabase-изменений | `WDZ_TEST_JWT='<access_token>' npm run security-check` | регресс RLS |

`access_token` брать из свежей браузерной сессии **обычного** участника, не
админа. Не сохранять токен в git, файлы, историю shell или чат; после проверки
закрыть shell/очистить команду из истории при необходимости.

## Момент 1 — Sonnet или Terra прислал отчёт по задаче

1. `npm run check` — все линтеры и repo-checks должны быть чистые.
2. Позови меня (Claude) на ревью — я прогоняю живьём через preview и сверяю с
   критериями приёмки задачи. Так было с T15.
3. Если задача трогала БД/RLS (T12, T14, T-SEC1…) — после применения миграции
   ещё и `npm run security-check`.

## Момент 2 — Применяю SQL-миграцию к self-host production

Порядок для любой миграции:
1. Сначала ревью SQL и backup, если migration очищает или переписывает данные.
   Никогда не накатывать весь `schema.sql` поверх production и не применять
   migration к старому cloud автоматически.
2. Скопировать одобренный файл в
   `/root/vibeclub/supabase/migrations/` на VPS и выполнить из
   `/root/vibeclub`:
   ```bash
   docker compose exec -T db psql -v ON_ERROR_STOP=1 -U postgres -d postgres \
     < supabase/migrations/<migration-file>.sql
   docker compose exec -T db psql -U postgres -d postgres \
     -c "NOTIFY pgrst, 'reload schema';"
   ```
   `ON_ERROR_STOP=1` обязателен: при первой SQL-ошибке команда должна завершиться
   неуспешно, а не продолжать файл.
3. Прогнать проверку из «шапки» файла migration (в каждом файле внизу есть блок
   «Проверка»). Для миграций безопасности это делает `npm run security-check` —
   не надо лезть в консоль браузера руками.

На момент T-CUTOVER 2026-07-15 все файлы, уже лежавшие в
`supabase/migrations/` (включая security-аудит 2026-07-14), отражены в
self-host production; `security-check` был зелёным. Следующая новая migration —
T-CONSENT — сначала проходит repo-review и отдельный live-гейт. Фактическое
состояние подтверждать SQL-проверкой из migration, а не этим историческим
абзацем.

## Про автоматизацию (Routines / cron)

Кратко: **часть проверок автоматизируется, часть — принципиально нет.**

- **Линтеры (`npm run check`)** — можно повесить на git-hook (пре-коммит), чтобы
  запускались сами. Когда захочешь — попроси, настрою через harness
  (это hook в настройках, не «память»).
- **`security-check`** — при включённой CAPTCHA требует готовый JWT обычного
  участника. Хранить токен в кроне или CI нельзя. Правильнее запускать руками
  после RLS-миграции и раз в месяц; токен брать из свежей браузерной сессии.
  ⚠️ С включённой капчей (SEC-10, 2026-07-14) парольный логин из скрипта
  отбивается — передавать готовый JWT сессии:
  `WDZ_TEST_JWT=<access_token> node scripts/security-check.mjs`
  (access_token — в браузере: localStorage → `sb-…-auth-token`).
- **Claude Routines (scheduled cloud agents)** — это запуск Claude по расписанию
  в облаке. Полезно для «раз в неделю собери лучшие проекты недели из витрины и
  накидай пост», «раз в день проверь, не упал ли деплой». Для безопасности —
  нет: там нужна интерактивная проверка результата, а не автопрогон. Заводить
  Routines имеет смысл, когда появится живая база и регулярный контент-ритм;
  сейчас рано. Когда дойдём — покажу пошагово.

Вывод: сейчас автоматизировать нечего, кроме, при желании, линтеров на пре-коммит.
Всё остальное — короткие ручные команды из шпаргалки выше, и я рядом на ревью.
