# 12 — Ранбук: когда что прогонять

Проект оброс проверками. Этот файл — единственное, что нужно держать под рукой:
что запускать, когда и зачем. Разложено по «моментам», а не по инструментам —
смотри свой момент и делай по списку.

## Быстрая шпаргалка

| Момент | Команда / действие | Зачем |
|---|---|---|
| После правок кода (Sonnet отчитался) | `npm run check` | линтеры JS+CSS = 0 |
| Перед коммитом | `npm run check` | не коммитить сломанное |
| После применения RLS-миграции | `npm run security-check <email> <пароль>` | привилегии не утекают |
| Перед анонсом | весь чек-лист `docs/05-launch.md` | ничего не забыто |
| Раз в месяц / после Supabase-изменений | `npm run security-check …` | регресс RLS |

`<email>/<пароль>` — **обычная** тестовая учётка (не твоя админская). Заведи одну
через сайт и держи её данные под рукой (не в git).

## Момент 1 — Sonnet прислал отчёт по задаче

1. `npm run check` — линтеры должны быть чистые (Sonnet это обещает в каждом промпте).
2. Позови меня (Claude) на ревью — я прогоняю живьём через preview и сверяю с
   критериями приёмки задачи. Так было с T15.
3. Если задача трогала БД/RLS (T12, T14, T-SEC1…) — после применения миграции
   ещё и `npm run security-check`.

## Момент 2 — Применил SQL-миграцию в Supabase

Порядок для любой миграции:
1. Supabase → SQL Editor → New query → вставить весь файл миграции → Run.
   «Success. No rows returned» = применилось (миграции меняют схему, строк не
   возвращают — это норма, не ошибка).
2. Прогнать проверку из «шапки» файла миграции (в каждом файле внизу есть блок
   «Проверка»). Для миграций безопасности это делает `npm run security-check` —
   не надо лезть в консоль браузера руками.

Применённые миграции на сегодня:
- `2026-07-03-stage-ask-profile.sql` — стадии/looking_for/skills (под T12–T14).
- `2026-07-06-rls-privilege-hardening.sql` — защита role/is_core (T-SEC1). ✅ применена.
- `2026-07-06-rls-privilege-hardening-fix.sql` — фикс функции (первая версия
  ломала легитимный edit из-за non-short-circuit AND). ✅ применена, security-check зелёный.

## Момент 3 — Готовлюсь к анонсу

Полный чек-лист — `docs/05-launch.md`. Не пропускать:
- миграция безопасности применена и `security-check` зелёный;
- витрина засеяна (`≥ 10–15` реальных проектов), тестовые вычищены;
- welcome-флоу проверен под реальной учёткой;
- прогон «глазами новичка с YouTube».

## Момент 4 — Периодически (без повода)

- `npm run security-check` раз в ~месяц и после любого изменения RLS/политик:
  ловит регресс, если новая миграция случайно открыла колонку.

## Про автоматизацию (Routines / cron)

Кратко: **часть проверок автоматизируется, часть — принципиально нет.**

- **Линтеры (`npm run check`)** — можно повесить на git-hook (пре-коммит), чтобы
  запускались сами. Когда захочешь — попроси, настрою через harness
  (это hook в настройках, не «память»).
- **`security-check`** — требует логина тестовой учёткой (email+пароль). Гонять
  это по расписанию в облаке = держать пароль в кроне, лишний риск. Правильнее —
  запускать руками в двух случаях: после RLS-миграции и раз в месяц. 30 секунд.
  ⚠️ С включённой капчей (SEC-10, 2026-07-14) парольный логин из скрипта
  отбивается — передавать готовый JWT сессии:
  `WDZ_TEST_JWT=<access_token> node scripts/security-check.mjs`
  (access_token — в браузере: localStorage → `sb-…-auth-token`).
- **Claude Routines (scheduled cloud agents)** — это запуск Claude по расписанию
  в облаке. Полезно для «раз в неделю собери лучшие проекты недели из витрины и
  накидай пост», «раз в день проверь, не упал ли деплой». Для безопасности —
  нет: там нужна интерактивная проверка результата, а не автопрогон. Заводить
  Routines имеет смысл, когда появится живая база и регулярный контент-ритм;
  сейчас рано. Когда дойдём — покажу пошагово.

Вывод: сейчас автоматизировать нечего, кроме, при желании, линтеров на пре-коммит.
Всё остальное — короткие ручные команды из шпаргалки выше, и я рядом на ревью.
