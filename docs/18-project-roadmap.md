# 18 — Дорожная карта до запуска и после него

**Статус на 2026-07-17.** Это канонический маршрут от текущего состояния до
публичного анонса и первых недель работы. История уже выполненных задач остаётся
в `04-tasks-sonnet.md` и хвосте `08-workflow.md`; здесь описано только то, что
ещё предстоит делать, в каком порядке и по каким признакам этап можно закрыть.

Если формулировки в других файлах расходятся с этим маршрутом, сначала сверить
фактический production и затем синхронизировать документацию. Launch-чекбоксы
живут в `05-launch.md`, юридические детали — в `14-ru-compliance.md`,
операторские проверки — в `17-operator-support.md`.

## Где проект находится сейчас

- Кодовая продуктовая база T0–T23 реализована.
- Self-hosted Supabase и Umami работают на RU-VPS; T-LOC/T-CUTOVER, финальный
  backup/restore и основной мониторинг закрыты 2026-07-15.
- T-CONSENT и T-CONSENT-RECONSENT закрыты 2026-07-17: актуальное согласие
  подтверждено, visual flash исправлен, финальная member JWT-проверка прошла
  29/29. Backfill и автоматическое удаление аккаунтов не выполнялись.
- Статический сайт пока обслуживается GitHub Pages/Fastly. До анонса origin
  должен переехать на RU-VPS в задаче T-FRONT-VPS.
- Индексация пока закрыта через `robots.txt`; открывать её можно только после
  финальной launch-приёмки.
- Supabase Studio доступен приватно одной командой через SSH-туннель по
  `17-operator-support.md`; публичного URL у Studio быть не должно.
- Последний агрегатный снимок production от 2026-07-17: 5 пользователей,
  5 профилей, 3 опубликованных проекта от одного автора, 1 комментарий,
  5 обращений (`done` — 2, `new` — 3). Это ориентир, а не вечная константа:
  перед решениями по запуску цифры обновлять без вывода PII.

## Критический путь

1. Уведомление РКН и отдельная оценка трансграничной передачи.
2. T-FRONT-VPS: подготовка, тестовый deploy, DNS-cutover, rollback-проверка.
3. Контент-гейт: 5–7 качественных проектов разных авторов и Telegram-ядро.
4. Финальная техническая, юридическая и продуктовая приёмка.
5. Открытие `robots.txt`, проверка `sitemap.xml`, зелёный production deploy.
6. Публичный анонс.
7. Операционный ритуал первых двух недель и решения по живым данным.

Контент и Telegram можно собирать параллельно этапам 1–2. T-IMG можно вести
параллельно только в отдельной ветке/worktree и не одновременно с другими
массовыми HTML-правками.

## Общие правила для всех этапов

- Не печатать и не сохранять в Git JWT, refresh token, service-role key,
  пароли, SSH-ключи, `.env`, адреса, телефоны, ИНН, email пользователей,
  UUID пользователей и другие PII/секреты.
- Перед production-действием делать read-only preflight, фиксировать baseline,
  способ проверки и план отката. Не расширять scope молча.
- Не удалять аккаунты или пользовательские данные без отдельного решения Тёмы,
  проверки связей и отзыва активных сессий.
- Старый cloud Supabase не является готовым rollback production. Не выполнять
  обратную синхронизацию, полный повторный импорт или backfill согласий.
- Для изменений схемы соблюдать RLS, запреты `PUBLIC EXECUTE`, безопасный
  `search_path`, server-side ownership checks и синхронизацию migration с
  `supabase/schema.sql`.
- Перед коммитом: `npm run check` и `git diff --check`; для изменённых JS ещё
  `node --check`. После push проверить соответствующий deploy.
- Ручную приёмку Тёмы и юридическое завершение не подменять техническим green.

## Этап 1 — уведомление РКН

### Цель

Подать уведомление об обработке персональных данных по фактической текущей
схеме, а не по историческому cloud-стеку. Отдельно инвентаризировать возможную
трансграничную передачу и определить с профильным специалистом, требуется ли
отдельное уведомление.

Официальные точки входа:

- уведомление об обработке: https://pd.rkn.gov.ru/operators-registry/notification/;
- трансграничная передача: https://pd.rkn.gov.ru/cross-border-transmission/.

РКН указывает, что уведомление можно направить на бумаге, с УКЭП или через
ЕСИА. Сведения об ответственном за организацию обработки, указанные в форме,
становятся общедоступными. Поэтому форму заполняет Тёма в своей сессии, а
Codex помогает только с фактической матрицей и проверкой формулировок. Это не
замена консультации юриста.

### Подготовить до открытия формы

**Codex проверяет по репозиторию и production без PII:**

- категории субъектов и данных;
- цели обработки, операции и способы обработки;
- какие системы участвуют: сайт, self-host Supabase, Storage, почта, Umami,
  резервные копии и операторские инструменты;
- российское размещение первичного production и технические меры защиты;
- фактические сроки/условия хранения и удаления, не выдумывая отсутствующие;
- дату начала обработки, которую должен подтвердить Тёма;
- фактические внешние браузерные запросы до и после T-FRONT-VPS.

**Тёма вводит только приватно в официальную форму:**

- данные ЕСИА;
- ИНН и статус оператора;
- регистрационный адрес/регион и телефон;
- точную дату начала обработки;
- юридические реквизиты Timeweb/ЦОД и иные поля, которых нет в публичных
  документах проекта.

Эти значения не пересылать в чат и не коммитить. Идентификатор, ключ записи,
PDF/скрин подтверждения хранить в приватном хранилище владельца, не в Git.

### Инвентаризация трансграничной передачи

Не копировать старый список «cloud Supabase + Resend + Umami Cloud»: он больше
не описывает production. Проверить реальные сетевые обращения в браузере и
текущую CSP. До T-FRONT-VPS участвуют GitHub Pages/Fastly; после cutover они
уйдут из активного origin. Как минимум отдельно оценить Google Fonts и
Cloudflare Turnstile, а также внешние аватары/ссылки, если они передают
пользовательские данные или сетевые идентификаторы иностранному получателю.

Техническая инвентаризация не равна юридическому выводу. Решение о составе и
моменте отдельного уведомления зафиксировать после сверки официальной формы и,
по возможности, с профильным юристом.

### Готовность этапа

- основное уведомление отправлено через официальный канал;
- Тёма сохранил подтверждение и приватные реквизиты вне Git;
- в `05-launch.md`, `08-workflow.md`, `14-ru-compliance.md` и этом файле
  зафиксированы дата и статус без публикации ключа/PII;
- трансграничная инвентаризация обновлена, решение и остаточный риск записаны;
- только после этого выдаётся отдельный исполнимый промпт T-FRONT-VPS.

## Этап 2 — T-FRONT-VPS

### Цель

Перенести `https://wedesignerz.com` с GitHub Pages/Fastly на Caddy текущего
RU-VPS, оставив GitHub репозиторием и источником deploy, но не публичным origin.

### Фаза A — безопасная подготовка репозитория

- Создать отдельный самодостаточный промпт для Terra с точным scope и
  запретами. До его ревью не менять DNS и production.
- Повторить allowlist/исключения текущей сборки `_site/`: `docs/`, `audits/`,
  `supabase/`, `infra/`, конфиги, скрипты и секреты не публикуются.
- Подготовить Caddy для apex и явное решение по `www`, внутренние 404,
  gzip/zstd, HTTPS и response headers: HSTS, `nosniff`, frame protection,
  Referrer-Policy, Permissions-Policy и CSP. Meta-CSP оставить как второй слой
  и не допускать расхождения с header CSP.
- Подготовить непривилегированного deploy-user, каталоги
  `/srv/wedesignerz/releases/<id>`, атомарный symlink `current`, ручной deploy
  и воспроизводимый rollback на предыдущий release.
- Секреты deploy и `known_hosts` хранить только в GitHub Environment. Root SSH
  из GitHub Actions не использовать. Первый workflow — только manual dispatch;
  автодеплой из `main` включать после живой приёмки.

### Фаза B — pre-cutover

- Собрать `_site/` и доказать, что внутренние файлы отсутствуют.
- Проверить все 18 HTML-страниц, assets, канонические ссылки, favicon/OG,
  корректный 404 и запрет directory listing.
- Проверить Auth, Storage, CAPTCHA, письма, recovery, Umami, CSP/CORS и чистую
  консоль на desktop и 375 px.
- Проверить deploy нового release и rollback на предыдущий без DNS-cutover.
- Снять baseline DNS/TTL и убедиться, что Pages остаётся рабочим путём отката.

### Фаза C — cutover и наблюдение

- Понизить TTL заранее, переключить apex/`www` на RU-VPS и проверить DNS/TLS.
- Подтвердить `curl -I`, 404 внутренних путей, весь e2e и отсутствие обращений
  к старому origin.
- Сохранить Pages как rollback-origin на 72 часа, но не публиковать разные
  версии сайта одновременно.
- В течение окна наблюдать health, Auth, Storage, Umami, backup и обращения.
  При критической регрессии откатить DNS по заранее проверенному плану.
- После стабильного окна убрать Pages/custom-domain как origin, обновить
  архитектуру, workflow, ранбуки и фактический статус. Старый cloud Supabase
  host из CSP убирать только после отдельной проверки rollback-границ.

### Готовность этапа

- DNS ведёт на RU-VPS, TLS и полный набор response headers подтверждены;
- deploy и rollback воспроизведены, root не участвует в CI;
- внутренние каталоги/файлы возвращают 404;
- Auth/Storage/Umami/CAPTCHA/почта и desktop/375 px зелёные;
- Pages больше не активный origin; документация соответствует production;
- `robots.txt` всё ещё закрыт.

## Этап 3 — контент и Telegram-ядро

Этот этап можно начинать сразу, не дожидаясь завершения РКН и T-FRONT-VPS.

### Витрина

- Launch-gate: 5–7 качественных реальных проектов разных авторов. Желательная
  стартовая цель остаётся 10–15.
- По снимку 2026-07-17 при сохранении трёх текущих проектов практический
  минимум — ещё 4 проекта от ещё 4 авторов: получится 7 проектов от 5 авторов.
- У каждого проекта: явное разрешение автора, хорошая обложка, понятное
  описание, реальный статус и отсутствие тестовых данных.
- Команда оставляет первый направленный комментарий на каждую новую
  публикацию; фейковые комментарии и метрики запрещены.

### Telegram

- Оформить чат: название, описание, правила, ссылка на сайт и закреп.
- Завести ядро 10–20 приглашённых участников, а не открывать пустой чат.
- Подготовить 3–5 стартовых тем и договориться, кто отвечает новичкам.
- Зафиксировать SLA модерации и ритуал «проект недели».

### Обращения

- До анонса разобрать все `feedback.status = new`; по последнему снимку их 3.
- Для диагностики использовать `17-operator-support.md`: сначала безопасный
  просмотр Studio, затем браузер/логи/агрегаты. Не менять данные «для проверки»
  без причины и не публиковать пользовательские сведения.

### Готовность этапа

- 5–7 проектов разных авторов прошли визуальную и содержательную приёмку;
- тестовый контент скрыт/удалён только после отдельного подтверждения владельца;
- Telegram-ядро и стартовые темы готовы;
- необработанных критичных обращений нет.

## Этап 4 — финальная launch-приёмка

Пройти на production desktop и 375 px полный путь нового пользователя:

1. главная → «Вступить»;
2. регистрация с двумя согласиями и Turnstile;
3. письмо → подтверждение → вход;
4. профиль и отдельное согласие на публичные контакты;
5. добавление проекта с обложкой/галереей;
6. админская модерация и публикация;
7. комментарий и редактирование/удаление своего комментария;
8. feedback и его разбор в админке;
9. recovery password;
10. отзыв dissemination: контакты исчезают, закрытый журнал сохраняет историю.

Дополнительно проверить:

- Umami-события воронки и UTM без PII;
- OG-превью живой ссылкой, canonical, favicon и sitemap;
- console/network без CSP/CORS/404/JS ошибок;
- response headers и недоступность внутренних путей;
- health/backup/диск/внешние мониторы;
- счётчики скрыты до порогов 30 участников и 15 проектов;
- текст приватности соответствует фактическому стеку и статусу уведомлений.

Все пункты `05-launch.md` должны быть закрыты доказательством или явной ручной
приёмкой Тёмы. Не переносить известный P0-дефект «после анонса».

## Этап 5 — открытие индексации

Это отдельный последний production-коммит перед анонсом:

- заменить закрывающий `robots.txt` на разрешающий индексацию;
- проверить `sitemap.xml`, canonical и отсутствие внутренних/служебных URL;
- дождаться зелёного VPS deploy;
- повторно проверить живые `robots.txt`, sitemap, главную и OG;
- не анонсировать, пока production не отдаёт именно этот release.

## Этап 6 — публичный анонс

- Довести видео по структуре из `05-launch.md` и использовать UTM-ссылку.
- Опубликовать видео, закреп в Telegram и ссылку на сайт синхронно.
- Команда должна быть готова к модерации, первым комментариям и обращениям.
- Не показывать фейковые счётчики и не обещать функции из бэклога.
- Зафиксировать время анонса и release SHA для последующей диагностики.

## Этап 7 — первые две недели

### Ежедневно

- новые регистрации, pending-проекты и `feedback.status = new`;
- Auth/Storage/Umami health, backup freshness, диск и внешние мониторы;
- соблюдение SLA модерации и первый полезный комментарий новым авторам;
- критичные ошибки в браузере и повторяющиеся обращения.

### Еженедельно

- воронка Umami: `join_click → auth_open → auth_success → submit_start →
  submit_sent → publish → comment_sent`;
- проект недели и подборка для Telegram/YouTube;
- причины незавершённой регистрации/submit и время до публикации;
- backup/restore дисциплина и разбор инцидентов без PII.

Пороговые значения в `09-growth-plan.md` пока гипотезы. Не менять продукт по
одному случаю; искать повторяющийся сигнал в данных и обращениях.

## Параллельные и последующие работы

### Можно делать параллельно до анонса

- **T-IMG:** WebP/responsive images, размеры и lazy loading; целевой вес
  статичных изображений — не более 4 МБ вместо примерно 11 МБ. Только отдельная
  ветка/worktree, полная визуальная проверка desktop/375 и без смешивания с
  T-FRONT-VPS.
- Стабилизационные drills: просроченный backup marker/404 и доставка алерта;
  перенос публичного anon key Auth-монитора из query string в header.
- Подготовка контента, Telegram и видео без открытия индексации.

### После стабильного анонса и появления данных

1. **T16:** спроектировать грейды v0/v1 только по живым данным.
2. Профиль/аккаунт: T-P2 → T-P1 → T-P3 → T-P4, если обращения подтверждают
   ценность; pretty URL зависит от username.
3. P2 из `09-growth-plan.md`: партнёры/клиенты, investor-срез, публичные
   метрики, OG проектов, retention-петля, кликабельные карточки и LCP/reduced
   motion.
4. Security follow-up из `16-security-status.md`: SHA-pin Actions, внешний
   `avatar_url`, RLS performance, eslint dependency, Node runtime и регулярный
   review образов/официального changelog self-host Supabase.
