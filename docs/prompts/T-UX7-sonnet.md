## A. Welcome-модалка — только при реальной регистрации

**Симптом:** участник со старым аккаунтом и опубликованным проектом вошёл
в аккаунт — и получил welcome-экран «Ты в клубе ✦» с шагами «покажи первый
проект».

**Диагноз:** `js/ui/authModal.js` — после ЛЮБОГО успешного входа welcome
показывается, если в `localStorage` нет ключа `wdz-welcome-shown`
(константа `WELCOME_KEY`, логика возле `showWelcome()`). Новый браузер /
устройство / очищенное хранилище = ложный welcome для старожила.

**Фикс:** сигнал «первый раз» брать из данных, а не из localStorage:
- после успешного входа получить профиль текущего пользователя
  (`profiles.created_at`; профиль создаётся триггером при регистрации);
- если возраст профиля < 10 минут — это регистрация, показать welcome;
- `wdz-welcome-shown` оставить как подавитель повторов (ставить после показа,
  проверять до запроса профиля — если стоит, не показывать и профиль не дёргать);
- ошибка запроса профиля → welcome НЕ показывать, вести себя как обычный вход.

**Приёмка:**
- вход существующим участником — welcome нет, обычное закрытие модалки с тостом;
- свежая регистрация — welcome есть;
- повторный вход в том же браузере сразу после регистрации — welcome не дублируется.

## B. Бургер-меню не должно переживать «Назад» (bfcache)

**Симптом (мобилка):** главная → открыть бургер → «Проекты» → системная
«Назад» → главная восстанавливается С ОТКРЫТЫМ меню.

**Диагноз:** браузер возвращает страницу из bfcache со снапшотом DOM
(открытый `.nav`, класс на `.header`, `aria-expanded="true"`).
Обработчика `pageshow` в `js/main.js` нет.

**Фикс:** в `js/main.js` (блок BURGER MENU) добавить:
`window.addEventListener('pageshow', (e) => { if (e.persisted) toggleMenu(false); })`.
Убедиться, что `toggleMenu(false)` корректно сбрасывает все состояния
(класс, aria-expanded, разблокировка скролла — см. пункт C).

**Приёмка:** эмуляция мобилки → открыть бургер → перейти по ссылке →
назад: меню закрыто, `aria-expanded="false"`, страница скроллится.

## C. Scroll lock: фон не скроллится под модалками

**Симптом:** при открытой auth-модалке колесо мыши / свайп скроллит
страницу под оверлеем.

**Фикс:**
- в `js/util.js` — утилита `lockScroll()` / `unlockScroll()` со счётчиком
  вложенности: `document.body.style.overflow = 'hidden'` + компенсация
  ширины скроллбара через `padding-right` (иначе контент прыгает),
  на разблокировке всё вернуть;
- применить в `js/ui/authModal.js` (открытие/закрытие оверлея) и в
  мобильном бургер-меню (`js/main.js`), если оно перекрывает страницу;
- проверить, нет ли других полноэкранных оверлеев (поиск по
  `overlay`/`modal` в `js/`); инлайн-подтверждения админки — не оверлей,
  не трогать.

**Приёмка:** открыта auth-модалка → колесо/свайп не двигает фон; закрыл —
позиция скролла та же, контент не прыгнул по ширине; Esc/крестик снимают лок.

## D. Аудит звёздочек ✦ — «один ✦ на зону»

**Симптом:** экран успеха submit: сверху маленький цветной глиф ✦
(`.submit-success-glyph`), а заголовок — «Улетело ✦». Два ✦ в одном экране.

**Правило:** в пределах одного экрана/модалки/блока — максимум один ✦.
Приоритет у глиф-элемента (маленький, цветной, отдельный); ✦ внутри
текстовых строк убирать.

**Где чистить (проверь все, список — отправная точка):**
- `js/i18n/ru.js`: `submit.success.title` «Улетело ✦» (рядом
  `.submit-success-glyph`), `auth.welcome.title` «Ты в клубе ✦» (в модалке
  уже есть `.auth-modal-glyph`) — из строк ✦ убрать;
- `auth.success.signin` / `auth.success.signup` / `me.save.success` —
  посмотреть, где рендерятся (тосты?): если рядом нет другого ✦, можно
  оставить; если есть — убрать;
- `admin.html` (`.adm-denied-glyph` + тексты рядом), `profile.html`
  (`.pf-not-found-glyph`), `404.html`, экраны «не найдено» — проверить на дубли;
- маркетинговые ✦ на главной (кикер витрины, how-terminal, member-label,
  футер «сделано во вайбкоде ✦») — это разные секции, дублями не считаются.

**Приёмка:** в отчёте — таблица «место → было → стало»; ни на одном
экране/модалке нет двух ✦ в одной зоне видимости.

## E. Изображения проекта: contain в просмотрщике + валидация пропорций

**Контекст:** просмотрщик галереи (`.pd-cover`, aspect-ratio 16/9,
`.pd-cover-img { object-fit: cover }`) кропит всё, что не ровно 16:9.
Пример: 1920×1007 (~1.9:1) — заметно обрезаны края. Пользователи будут
грузить скриншоты «как получится», готовых обложек у большинства нет.
Решение принято — три слоя защиты:

**E1. Просмотрщик галереи — показывать целиком:**
- `.pd-cover-img` → `object-fit: contain`, подложка контейнера —
  `var(--color-bg-sunken)` (уже стоит на `.pd-cover`);
- любые прошедшие валидацию пропорции (от квадрата до ультраширокого) —
  весь кадр виден, поля по бокам/сверху допустимы;
- мини-тумбы (`.pd-gallery-thumb`) — остаются cover (это превью);
- обложки карточек в витринах — не трогать (cover, паттерн YouTube).

**E2. Валидация пропорций на загрузке — только горизонтальные/квадратные:**
- правило: `width >= height` (квадрат допустим, вертикальное — нет);
- где: обе точки загрузки — обложка и изображения галереи в
  `submit.html`/`js/submit.js`, включая режим редактирования проекта
  (тот же форм-флоу); размеры изображения уже доступны в
  `js/image.js` (`optimizeImage` читает bitmap) — проверку логично
  встроить рядом, до ресайза;
- вертикальный файл: не добавлять в набор, показать ошибку у поля
  (строка в `js/i18n/ru.js`, тон вайб-рамки): «Только горизонтальные или
  квадратные изображения — вертикалка режется в витрине» (черновик,
  можно улучшить);
- EXIF-поворот учитывать (сравнивать стороны ПОСЛЕ нормализации
  ориентации, которую `optimizeImage` уже делает).

**E3. Хинт в форме заранее:**
- в подписи полей обложки и галереи добавить подсказку: «горизонтальные
  или квадратные, лучше всего 16:9» (i18n-строки рядом с существующими
  хинтами полей).

**Приёмка:**
- изображение 1920×1007 — в просмотрщике весь кадр с тонкими полями,
  без обрезки;
- квадрат 1000×1000 — целиком с полями по бокам;
- вертикальное 1080×1920 — не загружается ни как обложка, ни в галерею,
  ошибка с понятным текстом, форма живая (можно выбрать другой файл);
- тумбы галереи и карточки витрины — как раньше;
- хинт про пропорции виден в форме до выбора файла.

## F. Витрина на главной: мобильный лимит и кнопка снизу

Сейчас: `index.html` → `.showcase-grid` с `data-limit="8"`, ссылка
«Все проекты →» (`.showcase-all-link`) — сверху справа в `.showcase-head`.
На мобилке 8 карточек друг под другом = длиннющий скролл, а кнопка «Все
проекты» — до карточек, что нелогично.

**Решение (принято, вариант с горизонтальным скроллом отклонён):**
- на мобилке (breakpoint как у остальной мобильной верстки) показывать
  максимум 4 карточки; остальные скрывать CSS-ом
  (`nth-child(n+5)` в мобильном медиа-запросе — аккуратно, чтобы не задеть
  скелетоны), сеть не менять — `data-limit="8"` остаётся для десктопа;
- под сеткой добавить вторую ссылку «Все проекты →» (тот же стиль
  `.showcase-all-link`, модификатор), видимую ТОЛЬКО на мобилке;
  верхнюю на мобилке скрыть; десктоп — без изменений (8 карточек,
  ссылка сверху справа);
- новым классам с display:flex/grid, скрываемым через `hidden`, — сразу
  гард `.класс[hidden] { display: none; }` (повторяющийся баг проекта).

**Приёмка:** 375px — максимум 4 карточки, под ними «Все проекты →», сверху
ссылки нет; десктоп — как раньше (замерь до/после, ничего не сдвинулось).
