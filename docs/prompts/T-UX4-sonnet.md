## A. Фото — полная насыщенность всегда

Сейчас фото приглушены `filter: grayscale(...)` и «оживают» на hover. Тёма:
убрать, насыщенность всегда полная. Места (все в `styles.css`):
- `.portfolio-cover-img` — `grayscale(15%)` (~стр. 1049);
- `.team-photo` — `grayscale(20%)` (~стр. 1147);
- `.testimonial-photo` — `grayscale(20%)` (~стр. 1198);
- `.who-photo` — `grayscale(35%)` (~стр. 2111).
Убрать `filter: grayscale` из базовых состояний и `grayscale(0%)` из hover
(мертвеет). Hover-эффекты `transform: scale(...)` ОСТАВИТЬ. `transition`
почистить от filter, где он больше не нужен. Проверь grep-ом, что
`grayscale` больше нигде не приглушает контент (`-moz-osx-font-smoothing:
grayscale` — не трогать, это сглаживание шрифтов).

## B. «Уже с нами» — живая сетка вместо одной строки (`index.html`, блок `#community`)

Скрин Тёмы: одна маленькая marquee-строка ников в большом пустом блоке —
нелепо. Задача: зона участников красиво заполнена чипами с анимацией.
- Текущая реализация: `js/members.js` + `.member-marquee` (одна бегущая
  строка, T-D4). Переделать презентацию, источник данных тот же —
  реальные `profiles` из Supabase, никаких выдуманных ников.
- Целевая картина: **2–3 ряда чипов со смещением** (шахматное чередование —
  ряды сдвинуты друг относительно друга, не ровная таблица). Заполняют ширину
  зоны. При малом числе участников ряды добиваются повторами (как сейчас
  дублируется marquee-трек) — но подряд один и тот же ник не должен стоять
  рядом с собой.
- Анимация «живости» — выбери одно из двух (или скомбинируй, реши по красоте):
  1) ряды медленно едут по горизонтали в противоположных направлениях
     (разная скорость, бесшовный цикл — механика marquee уже есть);
  2) чипы периодически растворяются (fade out) и на их месте появляются
     другие ники (fade in), хаотично — раз в 2–4 сек один-два чипа.
- Чипы остаются кликабельными → `profile.html?id=…` (как сейчас).
- `prefers-reduced-motion: reduce` → статичные ряды без движения и ротации.
- Пустая БД → блок аккуратно скрыт/пустое состояние (не ломать текущее
  поведение).
- Мобилка: зона уже, рядов меньше/чипы мельче — без горизонтального скролла.

## C. Градиентная полоска шапки — узаконить и оживить

Разбор (важно понять): полоска под шапкой — сейчас **случайный баг**.
`.nav a::after` (подчёркивание пункта, styles.css ~195) имеет
`position: absolute; width: 0→100%`, но у `.nav a` НЕТ `position: relative`
(есть только у `.nav-home`). Поэтому ::after позиционируется от вьюпорта и
растягивается на всю ширину экрана. На подстраницах пункт «Проекты» имеет
`aria-current="page"` → полоса видна постоянно; на главной `aria-current`
у скрытого `.nav-home` → полосы нет, а hover любого пункта рисует её.
Тёме полоска нравится — делаем осознанно:
1. **Полоска — элемент шапки**: `.header::after` — 2px, `var(--gradient-electric)`,
   `background-size: 200% auto`, во всю ширину, на всех страницах по умолчанию
   (статичная).
2. **Оживление при hover интерактивов шапки**: когда курсор над ссылкой nav,
   кнопкой «Вступить»/«Выйти», именем пользователя или бургером — градиент
   полоски начинает течь (готовая `@keyframes gradient-flow` уже есть в
   styles.css). Реализация чистым CSS через `:has()`:
   `.header:has(a:hover, button:hover)::after { animation: gradient-flow …; }`
   (`:has` поддержан во всех актуальных браузерах; фолбэк не нужен — без
   поддержки полоска просто останется статичной).
3. **Убрать баговое подчёркивание**: удалить `.nav a::after`-механику
   (и `width:100%` для `aria-current`). Активная страница остаётся выделенной
   цветом (`.nav a[aria-current="page"] { color: … }` — уже есть). Проверь,
   что `.nav-home` на мобиле (где он виден в бургер-меню) не сломался.
4. `border-bottom` шапки (серый) можно убрать — полоска его заменяет; посмотри
   визуально, что лучше.
5. `prefers-reduced-motion` → полоска статична всегда.

## D. Скелетоны загрузки вместо «Загрузка…»

Скрин Тёмы: project.html при открытии — голый текст «Загрузка…» по центру
пустой страницы. Заменяем на скелетоны (плейсхолдеры по форме будущего
контента, с лёгким shimmer).
1. Общий CSS-примитив: `.skeleton` — блок цвета `var(--color-surface)` (или
   чуть светлее фона), `border-radius` по месту, shimmer — бегущий градиент
   (`@keyframes`), `prefers-reduced-motion` → без shimmer (статичный блок).
2. `project.html` (`.pd-loading`): скелет повторяет макет карточки — прямоугольник
   обложки 16:9, строка-заголовок, короткие строки меты, 2–3 строки описания.
3. `profile.html` (`.pf-loading`): круг аватара + строки имени/bio + пара
   карточек-заглушек.
4. Витрина (`js/showcase.js`, index + projects.html): пока идёт fetch — в гриде
   3–6 скелетон-карточек (обложка 16:9 + две строки), затем заменяются данными.
   Сейчас грид просто пустой с `aria-busy` — скелетоны добавить, `aria-busy`
   оставить.
5. Тексты «Загрузка…» из словаря можно оставить как скрытый a11y-текст
   (`visually-hidden` внутри скелетона) — скринридеру полезно.
6. Скелетон показывается только на время загрузки; ошибки/«не найден»/пустые
   состояния — как были.
