## Цель этапа

Подготовить проверяемую реализацию, в которой:

1. принятие политики при регистрации сохраняется в приватном серверно
   датированном журнале;
2. публичные контакты невозможно записать без отдельного активного согласия на
   распространение;
3. отзыв этого согласия одной транзакцией закрывает запись согласия и очищает
   все публичные контакты;
4. существующие контактные данные не считаются согласованными задним числом.

## 1. Версия политики

Введи понятную константу версии политики для регистрации, например
`privacy-2026-07-15-v1`, в подходящем модуле `js/`. Не разбрасывай строку по UI.
Та же точная строка должна проверяться на сервере в SQL.

Добавь статическую проверку в `scripts/`, включённую в `npm run check`, которая
падает при рассинхронизации версии между фронтом, migration и `schema.sql`.
Название и реализацию выбери сам, но проверка не должна обращаться к сети.

Если при работе изменяется содержание политики, версия должна быть осознанно
изменена до live-этапа. Клиентские `granted_at`, IP или user-agent как
доказательство не принимать и не собирать в этой задаче.

## 2. Приватный журнал `public.user_consents`

Добавь одну идемпотентную migration с датой задачи и синхронизируй итоговое
состояние в `supabase/schema.sql`.

Минимальная модель:

- `id uuid primary key default gen_random_uuid()`;
- `user_id uuid not null references auth.users(id) on delete cascade`;
- `consent_type text not null` с разрешёнными значениями `processing` и
  `dissemination`;
- `policy_version text not null`;
- `granted_at timestamptz not null default now()`;
- `revoked_at timestamptz null` с проверкой, что оно не раньше `granted_at`;
- `scope jsonb not null default '{}'::jsonb` и check, что это JSON-object.

Добавь индекс по `user_id` и partial unique index: у пользователя может быть не
более одной активной записи каждого `consent_type` (`revoked_at is null`).

RLS и права:

- RLS включён;
- `authenticated` читает только свои строки;
- `anon` не читает ничего;
- прямые `insert/update/delete` для `anon` и `authenticated` отсутствуют;
- пользователь не может подменить `user_id`, `consent_type`, версию или время;
- необходимые изменения идут только через ограниченные серверные операции;
- не раскрывай журнал через новый публичный view.

## 3. Регистрация и согласие `processing`

`signUpEmailPassword()` должен передавать в `options.data` только версию
политики рядом с существующим `display_name`. Не передавай клиентское время и
не передавай булево, которому сервер просто верит.

Расширь существующий `public.handle_new_user()` атомарно:

- точное значение версии читается из `new.raw_user_meta_data`;
- оно сравнивается с серверной разрешённой версией;
- при отсутствии/подмене версии функция бросает понятную машинную ошибку и вся
  регистрация откатывается, включая создание профиля;
- при совпадении создаются профиль и одна запись `processing` с `now()`;
- повторное выполнение не создаёт дубликат;
- пользовательские metadata не используются в RLS/authorization после этого.

Сохрани текущую логику имени/аватара и существующие security hardening/revoke.
Триггерную функцию нельзя вызывать напрямую через RPC: `revoke execute` от
`PUBLIC`, `anon`, `authenticated` должен остаться явным и в migration, и в
`schema.sql`.

## 4. Отдельное согласие `dissemination`

Scope активного согласия должен явно фиксировать список полей:

`telegram`, `website`, `github`, `phone`, `email_public`,
`custom_link_label`, `custom_link_url`.

Также зафиксируй в scope назначение публикации на публичной странице профиля.
Не включай туда логин-почту из `auth.users` и не автозаполняй `email_public`.

Реализуй минимальные RPC для авторизованного пользователя без параметра
`user_id` — идентификатор всегда берётся из `auth.uid()`:

- выдача/повторное подтверждение `dissemination` с серверным временем и точной
  версией;
- отзыв, который **в одной транзакции** ставит `revoked_at` активной записи и
  очищает все семь контактных колонок профиля;
- при отсутствии сессии RPC падают;
- повторная выдача при уже активном согласии идемпотентна;
- повторный отзыв безопасен и не восстанавливает данные.

Если RPC — `SECURITY DEFINER`, задай безопасный `search_path`, минимальную
логику и явные права:

- `revoke execute ... from PUBLIC, anon`;
- `grant execute ... to authenticated` только для RPC, которые действительно
  вызывает `/me`;
- любые внутренние/триггерные функции закрыты и от `authenticated`.

Не делай RPC, позволяющий читать или менять согласия другого пользователя.

## 5. Защита контактов на уровне БД

Одного чекбокса во фронте недостаточно. Добавь database trigger/проверку на
`profiles`, которая блокирует запись любого непустого контактного поля, если у
владельца профиля нет активного `dissemination` с текущей версией.

Требования:

- прямой REST `PATCH profiles` не обходит проверку;
- пустые строки нормализуются фронтом в `null`, но БД также не должна считать
  пробельную строку контактом;
- очистка контактов разрешена;
- профильные поля вне списка контактов продолжают сохраняться как раньше;
- логика не ломает административное восстановление из backup/SQL, но браузер
  не получает обхода;
- новая триггерная функция закрыта от прямого RPC-вызова.

В той же migration безопасно обнули существующие семь контактных колонок у
всех профилей. **Не создавай фиктивные согласия задним числом.** Перед live
применением `T-CONSENT-02` отдельно посчитает затронутые строки и создаст backup;
на этом repo-этапе SQL не запускается на production.

## 6. UX `/me`

В блоке контактов добавь отдельный контрол согласия со ссылкой на
`privacy.html#dissemination`.

Поведение:

- у пользователя без активного согласия контрол по умолчанию не отмечен;
- непустые контакты нельзя сохранить без явного включения;
- при включении сначала вызывается grant RPC, затем сохраняется профиль;
- активное состояние загружается из собственной записи журнала;
- снятие активного согласия вызывает revoke RPC, после успеха очищает поля в UI
  и не отправляет старые значения повторно;
- ошибки RPC показываются понятным i18n-текстом, кнопка не зависает;
- остальные поля профиля продолжают сохраняться;
- повторный submit идемпотентен.

Все новые пользовательские строки добавь в `js/i18n/ru.js`. Не хардкодь
русский в бизнес-логике. Для нового контейнера с `flex`/`grid`, скрываемого
через `hidden`, обязательно добавь явный `[hidden] { display: none; }` guard по
правилу проекта.

Юридическую формулировку чекбокса не расширяй от себя: используй нейтральный
технический текст о публикации **перечисленных контактных полей** и ссылку на
отдельный раздел. В отчёте явно пометь финальную формулировку как ручной гейт
Тёмы/юриста перед live-этапом.

## 8. Проверки

Расширь `scripts/security-check.mjs` без печати JWT/секретов. После будущего
применения migration скрипт с обычным `member` JWT должен проверить минимум:

1. `anon` не читает `user_consents`;
2. пользователь читает только свои consent rows;
3. прямой insert/update/delete журнала отбит;
4. прямой `PATCH profiles` с контактом без активного dissemination отбит;
5. grant RPC создаёт одну активную серверно датированную запись нужной версии;
6. после grant контакт сохраняется;
7. revoke RPC закрывает consent и очищает контакт;
8. повторный revoke безопасен;
9. попытка вызвать внутренние trigger functions через RPC отбита.

Тест обязан восстановить исходное состояние тестового профиля в `finally` или
явно сообщить, что не смог этого сделать. Не используй админскую учётку и не
создавай согласия/контакты для других пользователей.

На repo-этапе выполни:

- `npm run check`;
- `git diff --check`;
- статический аудит migration и `schema.sql` на совпадение объектов, policy,
  function grants и версии;
- локально через `python3 -m http.server 8080` — `/me`, auth-модалка и
  `privacy.html` на desktop и 375px, без горизонтального overflow и ошибок
  консоли.

`npm run security-check` на этом этапе не должен запускаться против старой
production-схемы: новые объекты ещё не применены. В отчёте пометь его как
ожидающий `T-CONSENT-02`.
