# T-CONSENT-02 — безопасное применение журнала согласий в production

Ты — Terra, исполнитель live-этапа P0-legal задачи We Designerz. Production
работает на self-hosted Supabase `api.wedesignerz.com`; доступ к RU-VPS —
`ssh vibeclub`, рабочая папка — `/root/vibeclub`.

## Жёсткий входной гейт

Не начинай live-изменения, пока в текущей задаче нет явного подтверждения Тёмы:

1. формулировки согласия при регистрации;
2. формулировки отдельного согласия на публикацию семи контактных полей;
3. полных реквизитов оператора для согласия на распространение. Текущая
   согласованная repo-формулировка: оператор — физическое лицо Андросов Артём
   Эдуардович; место жительства (преимущественного проживания) — Республика
   Индонезия, провинция Бали, округ Гианьяр, район Тегаллаланг, деревня Келики;
   контакт `veteristema@gmail.com`. Улица, дом и индекс намеренно не публикуются:
   приказ № 18 требует «место», но не устанавливает детализацию до полного
   адреса. Тёма подтвердил исходный адрес и поручил взять из него нужные данные;
4. что одобренный Тёмой способ идентификации участника реализован: отдельное
   непубличное ФИО передаётся в узкий grant RPC, подтверждённая почта берётся
   сервером из `auth.users`, оба значения сохраняются только в закрытом consent
   row и не попадают в публичный профиль;
5. что версия `privacy-2026-07-16-v4` синхронизирована между frontend, migration и
   `schema.sql` и соответствует утверждённому тексту;
6. что `T-CONSENT-UX` принята: непубличное полное имя вынесено выше и за
   пределы «Контактов», интерфейс объясняет требование закона, а текст не
   создаёт впечатления, что полное имя публикуется;
7. что `T-CONSENT-AUTH` принята: согласие на обработку ПДн и правила клуба —
   два самостоятельных unchecked-контрола, а регистрация явно проверяет оба.
8. что `T-CONSENT-VERSION-GATE` выполнена и принята: применённая историческая
   migration v2 восстановлена без изменений, v4 оформлена новой upgrade-
   migration, а grant RPC принимает и серверно проверяет версию, которую
   передал frontend. Пока этого нет, live-этап **запрещён**.

Версии `privacy-2026-07-15-v2` и `privacy-2026-07-15-v3` исторические;
live-этап с ними не начинать.

Если формулировка меняется — сначала измени `privacy.html`, i18n и версию
политики одновременно во frontend, migration и `schema.sql`; `npm run
check:consent` обязан подтвердить новую версию. Не подставляй вымышленные
реквизиты и не делай live-этап «временно» со старым текстом.

## Контекст и запреты

- `T-CONSENT-01` прошёл repo-review и локальную проверку на PostgreSQL 16.
- Read-only сверка production 2026-07-16 показала, что исходный вариант
  T-CONSENT уже был применён и выпущен с версией
  `privacy-2026-07-15-v2`: таблица, RLS, policy, RPC и DB-гейт существуют, а
  live frontend использует `v2`. Поэтому эта задача — **контролируемый апгрейд
  v2 → v4**, а не первичное создание схемы. Не считай само наличие объектов
  ошибкой и не повторяй migration вслепую без сверки baseline ниже.
- Snapshot на 2026-07-15 17:21 UTC, только для обнаружения drift: 4
  `auth.users`, 4 `profiles`, 0 профилей с контактами; в `user_consents` — одна
  активная `processing` v2 и две отозванные `dissemination` v2. Если к началу
  окна агрегаты изменились из-за легитимной активности, сначала объясни дельту
  и зафиксируй новый baseline; неизвестная дельта — стоп-гейт.
- Применять migration только к self-host production. Старый cloud не менять.
- Не импортировать cloud dump, не делать `truncate`, не трогать Storage,
  Umami, Caddy, DNS и `robots.txt`.
- Не печатать `.env`, JWT, anon/service-role secrets или значения контактов.
- Не считать старые контакты согласованными и не создавать backfill consent.
- Не удалять, не переписывать и не «повышать версию» у существующих consent
  rows. Исторические v2-записи должны сохраниться без изменений; migration
  меняет разрешённую сервером версию только для будущих grant/signup.
- Не смешивать T-FRONT-VPS, РКН или T-IMG с этим окном.

## Фаза A — repo-preflight

1. Покажи `git status --short` и отдели T-CONSENT diff от чужих изменений.
2. Убедись, что историческая migration
   `supabase/migrations/20260715091536_t_consent_01_user_consents.sql` совпадает
   с уже применённой v2 и **не будет запускаться повторно**. Проверь отдельную
   v4 upgrade-migration, созданную задачей `T-CONSENT-VERSION-GATE`: она
   содержит `begin`/`commit`, точную утверждённую версию, version-gate RPC и не
   обращается к cloud. Используй её точный путь из принятого repo diff.
3. Запусти `npm run check`, `git diff --check`, `node --check` для обоих
   check-скриптов.
4. Создай один осмысленный commit только из T-CONSENT-файлов. Пока не push.

## Фаза B — короткое окно и backup

1. Зафиксируй время начала окна. На VPS без вывода секретов временно поставь
   `DISABLE_SIGNUP=true` в `/root/vibeclub/.env`, пересоздай только `auth` и
   проверь health. Сохрани закрытую копию исходного `.env` с mode 600 только
   на время окна.
2. Запусти штатный `/root/vibeclub/scripts/backup.sh`; проверь свежесть DB и
   Storage-артефактов, не показывая их содержимое или credentials.
3. До migration выведи только агрегаты:
   - общее число `auth.users` и `profiles`;
   - число профилей, где заполнено хотя бы одно из семи контактных полей;
   - consent rows по типу, версии и active/revoked;
   - покрытие active `processing` по ролям профилей;
   - стабильный digest consent rows и строк с контактами без вывода PII или
     самих значений.
4. Если backup не свежий, auth unhealthy или состав затронутых строк
   неожиданен — migration не применять.

## Фаза C — атомарное применение

1. Передай ровно проверенную **v4 upgrade-migration** в `psql` по stdin. Не
   запускай повторно историческую v2 migration:

   ```bash
   ssh vibeclub "cd /root/vibeclub && docker compose exec -T db psql -v ON_ERROR_STOP=1 -U postgres -d postgres" \
     < supabase/migrations/20260715173135_t_consent_v4_upgrade.sql
   ```

2. Выполни `NOTIFY pgrst, 'reload schema';`.
3. Без PII проверь:
   - таблица, два индекса, RLS и policy существуют;
   - `authenticated` имеет только SELECT таблицы;
   - среди браузерных Data API ролей grant/revoke закрыты для `anon` и доступны
     только `authenticated`; стандартный привилегированный `service_role` может
     оставаться в ACL из Supabase default privileges, но не используется
     frontend и не отменяет обязательную проверку `auth.uid()`;
   - внутренние функции закрыты от `PUBLIC`, `anon`, `authenticated`;
   - версия БД совпадает с подготовленным frontend v4; live frontend до фазы D
     ещё ожидаемо остаётся v2, поэтому signups выключены;
   - grant RPC имеет только новую version-gated сигнатуру; вызов без версии или
     с v2/tampered version не создаёт consent row;
   - контакты прежних профилей очищены, фиктивных consent rows нет;
   - количество и digest всех существовавших до migration v2 consent rows не
     изменились.
4. Если `psql` завершился ошибкой, транзакция должна откатиться целиком.
   Остановись, проверь фактическое состояние и не повторяй вслепую.

## Фаза D — deploy frontend

1. Push подготовленного commit в `main`, дождись зелёного GitHub Pages deploy.
2. Проверь, что live `js/consent.js`, auth/signup, `/me` и

   смотрит только на self-host production.
3. До завершения живой приёмки signups остаются выключенными.

## Фаза E — живая приёмка

Используй отдельную обычную member-учётку без активного dissemination и без
контактов. JWT не печатай и не сохраняй в репозитории.

1. Запусти `WDZ_TEST_JWT='<access_token>' npm run security-check` — все старые
   и новые проверки зелёные; профиль восстановлен, созданная тестовая запись
   dissemination остаётся только в отозванной audit-истории.
2. В браузере desktop и 375 px проверь `/me`:
   - отдельный блок «Публикация контактов» расположен выше и вне «Контактов»;
   - полное имя явно помечено как непубличное, а требование отдельного согласия
     объяснено законодательством; однословное значение не принимается;
   - контакт без чекбокса не сохраняется;
   - grant → consent row содержит непустые `subject_full_name` и
     `subject_contact` без вывода их значений в отчёт, контакт сохраняется и
     виден на публичном профиле;
   - revoke → семь полей очищаются и исчезают с публичного профиля;
   - повторный revoke безопасен, консоль чистая.
3. Верни `DISABLE_SIGNUP=false`, пересоздай только `auth`, проверь health и
   Turnstile. Для CAPTCHA не обходи challenge: попроси Тёму пройти его.
4. Создай одну новую тестовую регистрацию через реальный UI, подтверди письмо,
   затем SQL-агрегатом проверь ровно одну активную `processing`-запись текущей
   версии. Удали тестовую учётку только по отдельному решению; удаление должно
   каскадно удалить её consent rows.
5. Повтори homepage/projects/login smoke, проверь Umami-событие и отсутствие
   новых ошибок auth/rest в логах.

## Фаза F — legacy-аккаунты, отдельный legal gate

Read-only сверка до окна показала: у одного admin и двух member-профилей нет
доказуемой active `processing`-записи, ещё у одного member есть только v2.
Миграция v4 не должна исправлять это backfill'ом.

1. После технической приёмки выведи только агрегаты покрытия, без email, ФИО,
   UUID и других идентификаторов.
2. Тёма отдельно классифицирует каждый legacy-аккаунт вне публичного отчёта:
   служебный/тестовый или реальный сохраняемый участник.
3. Тестовый аккаунт удалять только по отдельному явному решению Тёмы и после
   проверки связанных проектов/данных. Реальному сохраняемому участнику нельзя
   приписывать согласие задним числом: нужен явный re-consent текущей версии.
4. Если остаётся хотя бы один реальный аккаунт без доказуемого согласия v4,
   заведи отдельную repo-задачу `T-CONSENT-RECONSENT` на явный UI/API-flow и не
   отмечай legal gate закрытым. Не импровизируй массовый SQL INSERT/UPDATE.

## Rollback-границы

- До успешной migration: ничего не менять, исправить repo и открыть новое окно.
- Если migration откатилась: не push и не включать частичный обход.
- Если DB зелёная, а frontend deploy сломан: signups оставить выключенными,
  исправить/revert frontend; DB-гейт безопаснее старого поведения и не требует
  автоматического full restore.
- Full restore использовать только при доказанной порче данных и только из
  свежего проверенного backup в согласованном окне. Не перетирать новые записи
  вслепую.
- Закрытую копию `.env` удалить после подтверждённого восстановления исходного
  `DISABLE_SIGNUP=false`; секреты не включать в отчёт.

   `08-workflow.md`, `09-growth-plan.md`, `14-ru-compliance.md`,
   `16-security-status.md` и `docs/README.md`.
2. Отдельным docs-коммитом зафиксируй дату, SHA deploy, backup/restore proof,

3. Следующий gate — закрытие legacy-account inventory из фазы F, затем
   уведомление РКН. T-FRONT-VPS не начинать раньше него.
