## Контекст

Живые прогоны ловят баги (регистрация — T21, сброс пароля — 2026-07-12), но
единственный канал репорта — личный чат с Тёмой. Нужен встроенный путь: любой
человек — **включая гостя без аккаунта** (важный кейс: у него сломалась сама
регистрация) — описывает проблему на сайте, Тёма разбирает список в админке.

## Выход

### 1. Миграция БД: таблица `feedback`

Файл в `supabase/migrations/` с датой в имени (по образцу соседних),
идемпотентная (`create table if not exists`, `drop policy if exists` перед
`create policy`). Тот же DDL синхронизировать в `supabase/schema.sql`
(единый источник схемы чистой базы). Применяет Тёма руками в SQL Editor —
отметь это в ответе.

Колонки:

- `id uuid primary key default gen_random_uuid()`
- `created_at timestamptz not null default now()`
- `user_id uuid references auth.users(id) on delete set null` — nullable, гость = null
- `page text not null check (char_length(page) <= 300)` — путь страницы
- `message text not null check (char_length(message) between 10 and 2000)`
- `contact text check (char_length(contact) <= 200)` — nullable, «как с тобой связаться»
- `status text not null default 'new' check (status in ('new','done'))`

RLS (включить сразу):

- insert — `anon` и `authenticated`; `with check`: `user_id` либо null, либо
  `auth.uid()` (чужой uid подставить нельзя);
- select и update — только админ (`public.is_admin()`, функция уже есть
  в `schema.sql`); update админом — только поле `status` (достаточно политики,
  колонки не ограничиваем триггером);
- delete — никому (политики нет).

Серверный rate-limit в эту задачу НЕ входит — придёт единым паттерном из T18.
Анти-спам минимум: лимиты длины в БД (выше) + honeypot и cooldown на клиенте (ниже).

### 2. Клиент: ссылка в футере + модалка

- Во всех HTML-страницах (корень и `projects/`) в футер — ссылка
  «Нашли проблему?» рядом с существующими ссылками футера, тем же стилем.
  Если футер генерируется общим кодом — менять в одном месте.
- Клик → модалка (переиспользуй паттерн/стили auth-модалки, но это отдельный
  небольшой модуль, например `js/ui/feedbackModal.js`; подключение — через
  `js/main.js`, как остальные общие скрипты):
  - textarea «Что случилось?» — required, `maxlength="2000"`, подсказка-плейсхолдер
    в тоне `03-content.md` (на «ты», по-человечески);
  - поле «Как с тобой связаться? (необязательно)» — `maxlength="200"`,
    для залогиненного можно не показывать (есть `user_id`);
  - honeypot: скрытое поле-ловушка (visually hidden, не `display:none` от
    автозаполнения ботов), заполнено → молча «успех» без insert;
  - `page` подставляется автоматически: `location.pathname + location.search`;
  - `user_id` — из текущей сессии, если есть; гость шлёт с null;
  - сабмит → insert через существующий клиент `js/supabase.js`; успех →
    состояние «Спасибо, разберёмся» внутри модалки; ошибка — человеко-читаемо
    через строки в `js/i18n/ru.js` (все новые тексты — туда);
  - клиентский cooldown: после успешной отправки повторная — не раньше чем
    через 60 сек (кнопка disabled с отсчётом, как resend в auth-модалке).
- Помни правило проекта: новый класс с `display:flex`/`grid`, скрываемый через
  `hidden`, обязан иметь гард `.класс[hidden] { display: none; }`.

### 3. Админка: блок «Обращения»

`admin.html` + `js/admin.js`, по образцу существующих блоков модерации:

- список `status='new'` (свежие сверху): дата, страница, текст, контакт,
  почта/имя автора если `user_id` есть (join к `profiles` — как уже делается
  в админке);
- кнопка «Обработано» → `update status='done'`;
- переключатель/вкладка «показать обработанные» — по желанию, если дёшево;
- пустое состояние: «Обращений нет — тишина и покой».

## Границы

- Не трогать auth-логику, витрину, submit.
- Никакого `service_role`, только `anon`-ключ; защита — RLS.
- Email-уведомления Тёме — НЕ в этой задаче (нет сервера; вернёмся после T19).

## Приёмка

- Гость (разлогинен) отправляет репорт с главной — запись в БД с `user_id=null`.
- Залогиненный отправляет со страницы проекта — запись с его `user_id`,
  `page` заполнен.
- Сообщение из 5 символов не уходит (клиент), из 3000 — отбивает и БД
  (проверка insert из консоли под обычной учёткой).
- Не-админ через консоль не может `select` из `feedback` (0 строк) и не может
  `update` чужой статус.
- Админ видит список в `admin.html`, «Обработано» убирает запись из new.
- Honeypot-заполненная отправка не создаёт строку.
- 375px ок; консоль чистая; линтеры 0; `npm run security-check` зелёный.
- В ответе — напоминание Тёме применить миграцию в SQL Editor.
