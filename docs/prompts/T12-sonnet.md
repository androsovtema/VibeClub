бэкенд Supabase. Прочитай перед стартом: `CLAUDE.md`, `docs/08-workflow.md`,

Суть: карточка проекта не отвечает «на какой стадии проект?» и «что автору нужно?».
Проект должен быть не витринным экспонатом, а точкой входа: посмотрел → понял, чем
помочь → включился. Это же сырьё для будущих грейдов (`docs/10-membership.md`).

## A. Словарь значений (единый источник)

БД принимает только ключи (check-констрейнты уже в миграции):
- `stage`: `idea | prototype | mvp | users | commercial`
- `looking_for` (text[]): `feedback | testers | designer | developer | cofounder | client | investor`

Сделай модуль словаря (например, `js/vocab.js`): массивы ключей + подписи через
`js/i18n/ru.js`. Черновик подписей (финалит Тёма, тон — вайб-рамка):
- stage: идея · прототип · MVP · есть пользователи · зарабатывает
- looking_for: фидбек · тестеры · дизайнер · разработчик · кофаундер · клиенты · инвестиции
В БД — ключи, в UI — русские подписи. Свободный ввод невозможен.

## B. `submit.html` + `js/submit.js`

- Селект/чипы «Стадия» (один, необязательный) + мультивыбор-чипы «Что ищу»
  (необязательные) — в стиле существующих чип-групп тегов/инструментов.
- Подписи полей человечные, без анкетности. Черновик: «На какой стадии?» и
  «Чего не хватает?» (хинт: «отметь — и в обсуждение придут те, кто может помочь»).
- Insert в `projects`: `stage` (или null), `looking_for` (массив ключей, может
  быть пустым).

## C. Карточка витрины (`js/projects.js`, `renderProjectCard`)

- Бейдж стадии в мета-строке карточки (моно, тихий — не конкурирует с названием).
- Если есть `looking_for` — компактный маркер первого запроса (например,
  «ищет: тестеры»). Не перегружать: максимум один запрос на карточке.
- Пустые значения не рендерят пустых элементов.

## D. `project.html` + `js/project.js`

- Стадия — в мета-строке (рядом с датой), моно-бейдж.
- Блок «Автор ищет» с чипами всех запросов — между описанием и кнопкой действия
  либо рядом с тегами (по месту, аккуратно). Если `looking_for` пуст — блока нет
  совсем (и гард `[hidden]`, если скрываешь через hidden! см. CLAUDE.md).

## E. `projects.html` + `js/showcase.js`

- Фильтр-чипы «ищут» по `looking_for` (contains, `.cs.{key}` в PostgREST — либо
  `overlaps`, выбери и проверь). URL-параметр `?looking=<key>` по аналогии с
  существующими `?cat=` и `?tool=` (см. T-UX2/T-UX3 — не сломай их).
- `fetchPublishedProjects` расширяется параметром `looking`.

## Ограничения

- Все строки — в `js/i18n/ru.js`, тон — «Вайб-рамка» из `03-content.md`
  (никакой машинной анкетности: не «выберите значение», а по-человечески).
- Токены из `css/tokens.css`. Новые flex/grid-классы, скрываемые `hidden`, —
  сразу гард `[hidden]{display:none}` (ловили 4 раза).
- XSS: значения из БД валидируй по словарю (ключ не из словаря — не рендерить),
  подписи — только через `t()`; `textContent`/`createElement`.
- Не ломать: фильтры `?cat=`/`?tool=`, сортировку, auth-aware `data-join`,
  карточку T-D3, skeleton-состояния (T-UX4).

## Проверка (критерии приёмки)

- Проект без стадии/запросов выглядит ровно как раньше (ничего пустого).
- Submit: отправка со стадией и 2 запросами → в БД ключи; без них → null/`{}`.
- Карточка и страница проекта показывают русские подписи; мусорный ключ в БД
  (вручную подставить в ответ) не рендерится.
- `projects.html?looking=testers` фильтрует; комбинация с `?cat=` не конфликтует
  (реши: последний выигрывает или совместно — задокументируй в отчёте).
- Мобилка 375px: чипы submit и блок «Автор ищет» не разъезжаются.
- Консоль чистая; `npm run lint:js` / `npm run lint:css` — 0.
