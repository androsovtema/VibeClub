## A. Кнопки действий у комментов — стилизовать (project.html)

Скрин Тёмы: «Изменить»/«Удалить» выглядят как нестилизованные браузерные кнопки
(рамка, светлый фон). Причина: в `styles.css` нет глобального сброса `button`,
а `.comment-action-btn` задаёт только font-size/color — background/border
остаются дефолтными.
- Привести `.comment-action-btn` к виду сайта: прозрачный фон, без рамки (или
  тонкая рамка-чип в духе `.filter-tag`), моно/мелкий текст, hover уже есть.
  Тот же вид у «Да/Отмена» подтверждения удаления и согласованный компактный
  вид у «Сохранить/Отмена» режима редактирования (сейчас там полноразмерные
  btn-primary/btn-secondary — слишком тяжело для инлайн-действия; «Сохранить»
  НЕ должен быть градиентным primary, см. раздел D).
- «Всё слиплось»: добавить воздух между кнопкой «Отправить» формы и первым
  комментом (отступ у `.comments-list` сверху, ~`--space-sm`), и проверить
  вертикальный ритм внутри реплики (имя → текст → действия).
- Глобальный сброс `button` НЕ вводить (перелопатит все страницы) — точечно
  стилизуй классы.

## B. Бейджи инструментов кликабельны → витрина с фильтром по инструменту

Тёма: «хочу кликнуть на Claude и посмотреть другие проекты, сделанные с ним».
1. `js/projects.js`: `fetchPublishedProjects({ ..., tool })` — при наличии
   `tool` добавлять `query.contains('tools', [tool])`.
2. `js/showcase.js` (управляет витриной на `projects.html` и главной):
   - на `projects.html` при загрузке читать `?tool=X` из URL
     (`new URLSearchParams`), класть в state, передавать в fetch;
   - активный tool-фильтр показать визуально: чип «✦ X ✕» (или похоже) рядом с
     чипами категорий; клик по нему — сброс фильтра (state.tool = null, убрать
     параметр из URL через `history.replaceState`, перезагрузить ленту);
   - tool-фильтр комбинируется с категориями и сортировкой (это просто ещё один
     `.contains` в запросе).
3. `js/project.js`: бейдж `.pd-tool-badge` рендерить ссылкой
   `<a href="projects.html?tool=<encodeURIComponent(tool)>">` (значение из БД —
   экранируй при вставке в атрибут через encodeURIComponent, текст —
   `textContent` как сейчас). Hover — видимая интерактивность (подчёркивание/
   смена цвета).
4. Бейдж инструмента в углу обложки на карточках витрины НЕ делать ссылкой
   (клик по карточке ведёт на проект — не создавать вложенные клик-цели).

## C. Сортировка: селект → сегмент-контрол

Скрины Тёмы: закрытый селект — шеврон прилип к правой границе; открытый —
нативное браузерное меню, чужое для тёмного стиля. Нативный дропдаун не
стилизуется кроссбраузерно — заменяем контрол:
- Вместо `<select data-sort>` — сегмент-контрол из двух кнопок
  «Свежие | Популярные» (вид — как активный/неактивный `.filter-tag`, или
  слитная «пилюля» из двух сегментов; выбери что чище в текущем стиле).
- `js/showcase.js`: обработчик `change` селекта → клики по кнопкам
  (`state.sort = 'new' | 'top'`, перезагрузка ленты, класс active).
- `aria`: группа с `role="group"` + `aria-pressed` на кнопках (или
  radio-паттерн) — доступность не хуже селекта.
- Применить на обеих страницах, где есть сортировка (проверь главную:
  если там селекта нет — только `projects.html`).
- Подпись «Сортировка» можно оставить или убрать — реши по месту.

## D. Иерархия кнопок: один primary на экран

Тёма: «Открыть проект» и «Отправить» выглядят одинаково — непонятно, что
главнее. Правило: **на экране один градиентный `btn-primary` — главное действие
страницы**.
- `project.html`: primary остаётся у «Открыть проект →». Кнопка «Отправить»
  формы коммента → вторичная (обводка/поверхность, компактнее). «Войти» в
  гейте комментов → тоже вторичная (гость и так видит primary «Открыть проект»).
- `submit.html`: не трогать — там «Отправить на модерацию» единственный primary,
  это правильно. На экране успеха «Добавить ещё один» primary + «Чат сообщества»
  secondary — уже ок.
- Проверь `index.html`/`projects.html` на дубли primary в одном вьюпорте — если
  найдёшь явные (кроме hero, где primary один), отметь в отчёте, чинить не надо.

## Ограничения

- Токены, i18n (`js/i18n/ru.js`) для новых строк (чип сброса фильтра, подписи
  сегментов — переиспользуй существующие ключи опций сортировки, если есть).
- Пользовательские данные — `textContent`/`encodeURIComponent`; название
  инструмента приходит из БД, в URL и обратно из URL — не доверять, в fetch
  идёт как параметр `.contains` (Supabase сам параметризует), в DOM — только
  textContent.
- Помни правило `display:flex` vs `[hidden]` из `CLAUDE.md`.
- Не трогать: логику комментов T-UX1 (edit/delete), auth-aware CTA, шапку.

## Проверка (критерии приёмки)

- Комменты: «Изменить/Удалить/Да/Отмена/Сохранить» выглядят по-сайтовому
  (никакого браузерного дефолта), между формой и лентой есть воздух.
- `project.html` → клик по бейджу «Claude» → `projects.html?tool=Claude`,
  в витрине только проекты с Claude, виден чип активного фильтра, клик по ✕
  сбрасывает (URL чистится без перезагрузки).
- `?tool=несуществующий` → честное пустое состояние витрины.
- Сортировка: сегмент-контрол переключает свежие/популярные, селекта больше
  нет; клавиатурой доступно.
- `project.html`: один градиентный primary («Открыть проект»), «Отправить» и
  «Войти» гейта — вторичные.
- Мобилка 375px: сегмент-контрол и чипы не разъезжаются, без горизонтального
  скролла.
- Консоль чистая; `npm run lint:js` / `npm run lint:css` — 0 ошибок.
