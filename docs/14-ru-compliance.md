## Локализация: выбранная схема (ресерч 2026-07-13)

Прямого российского BaaS-аналога Supabase нет. Amvera обещала managed
Supabase к 30.06.2026 — на 13.07.2026 статус «в разработке, 50%», срок сорван
(следить: `amvera.ru/marketplace`, если выйдет раньше нашего переезда —
пересмотреть, снимет ops-нагрузку). Пересборка на Yandex Cloud / VK Cloud
managed Postgres означает переписывание всей связки PostgREST+GoTrue+RLS —
отклонено. PocketBase/Appwrite — другая API-модель, переписывать весь фронт —
отклонено.

**Реализовано: self-hosted Supabase на российском VPS.**

- Один VPS 4–8 ГБ RAM (Timeweb Cloud — дешевле, ~1500 ₽/мес за 4 ГБ;
  Selectel — дороже, стабильнее). Прод-минимум Supabase — 4 ГБ; наши
  контейнеры: db, kong, auth (GoTrue), rest (PostgREST), storage + imgproxy,
  meta. Realtime, edge functions и logflare-аналитику не используем — отключить,
  влезем в 4 ГБ с запасом.
- Фронт не меняется: только `SUPABASE_URL` + `anon`-ключ в `js/config.js`.
  Данные: pg_dump/restore (auth.users с хэшами паролей переносится), файлы
  storage перелить, на текущем объёме (3 юзера) — тривиально.
- **Почта: Resend → Unisender Go Web API** через локальный `mail-bridge` на
  VPS. Прямой SMTP оказался недоступен; фактическая схема и проверки описаны
  в шаге 6 `infra/RUNBOOK.md`. До анонса обязательны SPF/DKIM, верификация
  домена и живой тест на новом адресе.
- **Umami Cloud → self-host Umami на том же VPS** (лёгкий: node + postgres,
  сотни МБ). Меняются только `src` скрипта и id — оба уже параметризованы в
  `js/analytics.js`/`js/config.js`. Метрика отклонена: cookies + тяжёлый скрипт.
- Статика временно остаётся на GitHub Pages, затем до анонса переезжает на тот
  же RU-VPS (T-FRONT-VPS). Это отдельный cutover после БД и уведомления РКН.
- Ops-обязанности после переезда: ночной pg_dump в S3-совместимое хранилище
  того же провайдера, обновления docker-образов раз в месяц, мониторинг
  диска/RAM. Это цена self-host — заложить в ритуал.

**Бонус варианта Б (уточнён по внешнему аудиту 2026-07-14):** после переезда
серверная обработка ПДн (БД, почта, аналитика) — в РФ. После T-FRONT-VPS
статика тоже будет отдаваться из РФ. Но браузер всё ещё обращается как минимум
к Google Fonts и Cloudflare Turnstile — а политика относит IP к ПДн. Поэтому
вопрос трансграничной передачи НЕ считать закрытым автоматически: проверить
финальную схему перед подачей (self-host шрифтов — простой способ убрать одну
зависимость; замену Turnstile решать отдельно).

**Сроки подачи (важно):** обработка фактически уже идёт — в базе живые
пользователи, а РКН требует уведомление **до начала** обработки. Уведомление
не откладывать под переезд фронта/статики. Cutover закрыт 2026-07-15; после
T-CONSENT подать по фактической российской схеме.

## Как подавать уведомления РКН: пошагово (обновлено 2026-07-14)

### Решение по ЦОД

В форме уведомления об обработке есть обязательное поле «сведения о
местонахождении базы данных с ПДн граждан РФ»: адрес ЦОД + реквизиты
провайдера. Наша production-БД с 2026-07-15 — RU-VPS Timeweb. Развилка ниже
сохранена как историческое обоснование решения:

- **Вариант А (подать сейчас, честно указать Германию).** Плюс: закрыт штраф
  за неуведомление, добросовестность. Минус: уведомление с иностранным ЦОД —
  самодонос о нарушении локализации, вероятен запрос РКН с требованием устранить.
- **Вариант Б (сначала локализация, потом уведомление).** Перенести БД на
  RU-VPS (этап 1 приблизить), в форме указать российский ЦОД — уведомление
  чистое. Минус: пока не переехали, ходим без уведомления (штраф физлицу
  5–10 тыс. ₽, ловят обычно по жалобам).
- Лгать в форме (вариант «указать РФ, оставаясь во Франкфурте») — нельзя,
  это хуже обоих.

**Решение Тёмы (2026-07-13): вариант Б** — сначала локализация, потом
уведомление. Локализация завершена 2026-07-15. Это не отменяет просрочку:
следующий юридический шаг после T-CONSENT — подача уведомления с реквизитами
российского ЦОД.

### Уведомление об обработке ПДн

1. `pd.rkn.gov.ru` → «Реестр операторов» → «Электронные формы заявлений» →
   «Уведомление о намерении осуществлять обработку персональных данных».
2. Подписание: удобнее всего через Госуслуги (ЕСИА); альтернативы — ЭЦП или
   распечатать и отправить почтой в территориальный орган РКН.
3. Оператор: **физическое лицо** — Андросов Артём Эдуардович,
   `veteristema@gmail.com`. Публичная политика указывает текущее место
   преимущественного проживания на Бали. Для уведомления РКН отдельно взять
   ИНН, регистрационный адрес, регион и телефон из актуальных документов/ЕСИА:
   административная форма и публичный текст согласия — разные контуры, значения
   для формы не выдумывать и автоматически не публиковать на сайте.
4. Готовые формулировки полей:
   - **Цель обработки:** регистрация и идентификация пользователей
     интернет-сообщества, ведение публичных профилей участников, публикация
     пользовательских проектов, обратная связь и поддержка пользователей.
   - **Правовое основание:** согласие субъекта ПДн (ст. 6 ч. 1 п. 1 152-ФЗ);
     сама подача — ст. 22 152-ФЗ.
   - **Категории ПДн:** фамилия/имя или псевдоним, адрес электронной почты,
     номер телефона (по желанию пользователя), фотография (аватар),
     технические данные (IP-адрес).
   - **Категории субъектов:** пользователи сайта `wedesignerz.com`.
   - **Действия с ПДн:** сбор, запись, систематизация, накопление, хранение,
     уточнение, извлечение, использование, передача (предоставление, доступ),
     блокирование, удаление, уничтожение.
   - **Способ обработки:** автоматизированная, с передачей по сети Интернет.
   - **Меры защиты:** шифрование канала (TLS), разграничение доступа на уровне
     СУБД (row-level security), парольная аутентификация, резервное копирование,
     политика конфиденциальности на сайте. Ответственный за организацию
     обработки — сам оператор (Тёма).
   - **Местонахождение БД:** российский ЦОД Timeweb, в котором размещён текущий
     RU-VPS. Точный адрес ЦОД, название юрлица, ИНН/ОГРН и адрес провайдера
     взять из актуального договора/кабинета Timeweb перед подачей; исторические
     реквизиты AWS/Supabase не копировать.
5. После отправки РКН пришлёт **идентификатор и ключ записи** — сохранить,
   без них не подать изменения. В реестр вносят до 30 дней; проверка — поиск
   по реестру операторов на том же сайте.
6. Любые изменения (смена ЦОД при переезде на RU-VPS!) — «информационное
   письмо о внесении изменений» до 15-го числа следующего месяца.

### Уведомление о трансграничной передаче

Отдельная форма там же («Уведомление о намерении осуществлять трансграничную
передачу персональных данных»). Подаётся один раз, до начала передачи.

- **До cutover** фактическая схема включает Supabase/AWS в Германии, Resend и
  Umami Cloud; если уведомление подаётся до переезда, указывать её честно.
- **После DB cutover** не копировать старый список автоматически. До
  T-FRONT-VPS браузер ещё обращается к GitHub Pages/Fastly; после него остаются
  Google Fonts и Cloudflare Turnstile. Сначала зафиксировать, какие ПДн и кому
  реально передаются, затем заполнить страны и получателей по факту.
- Для страны вне перечня достаточной защиты учитывать срок ожидания после
  подачи и право РКН ограничить передачу. Конкретный перечень и реквизиты
  получателей перед отправкой формы перепроверить по актуальным договорам и
  официальным сведениям сервисов.

### После подачи — постоянные обязанности

- Утечка ПДн: уведомить РКН в течение **24 часов**, результаты внутреннего
  расследования — в **72 часа**.
- Политика на сайте и уведомление не должны расходиться (категории, цели).
- Дата ревизии этого дока — при любом изменении стека или раз в полгода.

Инфраструктура RU-VPS уже поднята, поэтому безопаснее завершить cutover до
привлечения новой аудитории. T-CUTOVER и стабилизация закрыты 2026-07-15.
Актуальная последовательность: **T-CONSENT → уведомление РКН → T-FRONT-VPS →
контент-гейт → снять `robots.txt` → анонс**. T-IMG можно вести параллельно.
Источник статуса — хвост `docs/08-workflow.md`.
